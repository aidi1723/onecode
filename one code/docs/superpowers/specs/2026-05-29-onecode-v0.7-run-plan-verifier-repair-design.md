# OneCode v0.7 Run-Plan Verifier Repair Design

## Goal

Add an explicit, model-assisted repair loop for `run-plan` when controlled verifiers fail. The repair loop uses verifier failure evidence to request patch-only repairs, applies those patches through the existing guarded kernel path, reruns selected verifiers, and records all evidence in the ledger.

This is a local-development feature. It does not publish code, push branches, or create PRs.

## Scope

In scope:

- `run-plan` only.
- Controlled verifier failures from `--verifier-policy` and `--verifier`.
- Explicit opt-in repair flags.
- Patch-only model repair plans.
- Ledger and inspect evidence for repair attempts.

Out of scope:

- `run`, `run-model`, execution-plan, and TUI repair.
- Arbitrary shell execution.
- Asset-generating repairs.
- Automatic repair without an explicit CLI flag.
- Multiple model providers beyond the existing provider configuration machinery.

## CLI Surface

Extend `run-plan` with:

- `--repair-model MODEL`
- `--repair-provider PROVIDER`
- `--repair-endpoint URL`
- `--repair-api-key KEY`
- `--max-repair-attempts N`

Repair is enabled only when `--max-repair-attempts` is greater than zero and a verifier fails. If repair is requested without selected verifiers, the CLI returns a parser error before writes.

## Repair Prompt

The repair prompt must include evidence, not authority:

- original task
- planned asset paths
- failed verifier id
- verifier status, reason, exit code, stdout tail, stderr tail
- task status code and transition fields
- task resume decisions when present
- current run id and workspace-relative affected paths

The prompt must require JSON with patches only. It must reject assets and execution plans.

## Repair Plan Validation

Use the existing `ModelPlan` validation path. A repair plan is accepted only when:

- it has at least one patch
- it has no assets
- it has no execution plan

Rejected repair plans halt the repair loop and record `repair_rejected_reason=repair_plan_must_use_patches_only`.

## Execution Flow

For `run-plan` with selected verifiers and repair enabled:

1. Execute the plan through the existing guarded write path.
2. Run selected controlled verifiers.
3. If all verifiers pass, return deliverable result with no repair.
4. If a verifier fails, build an evidence-based repair prompt.
5. Ask the configured model provider for a patch-only plan.
6. Apply patches through existing `execute_model_plan()` and `patch_text` behavior.
7. Rerun the selected verifiers.
8. Repeat until verifiers pass or `max_repair_attempts` is exhausted.
9. Write final ledger evidence.

Repair attempts must never bypass `LogosGate`, `PathGuard`, patch syntax checks, checkpoint, ledger, or `IchingKernel` status projection.

## Ledger Evidence

Add these fields when repair is attempted:

- `repair_attempt_count`
- `repaired`
- `initial_verifier_results`
- `repair_verifier_results`
- `repair_results`
- `repair_rejected_reason`
- `repair_prompt_evidence`

`inspect` should surface these fields when present.

## Error Handling

- Missing model API key: parser error before repair attempt; asset writes and verifier evidence remain recorded.
- Repair plan contains assets or execution plan: halt repair and record rejection.
- Patch application fails: record repair result and stop when `IchingKernel` dispatch says stop.
- Verifier still fails after all attempts: final result remains halted/blocked with latest verifier evidence.
- Verifier passes after repair: final result becomes completed/deliverable and records `repaired=true`.

## Testing

Implementation must be test-first. Required tests:

- verifier failure triggers one patch-only repair attempt and reruns verifier successfully.
- repair plan with assets is rejected and recorded.
- exhausted repair attempts leave final result halted with latest verifier evidence.
- repair is not attempted when `max_repair_attempts=0`.
- repair requested without verifiers fails before writes.
- inspect reports repair evidence.
- denied `bash_execution` and `execute_pytest` remain denied.

## Acceptance Criteria

- `run-plan` can repair a verifier failure through patch-only model output.
- Repair is opt-in and bounded by `max_repair_attempts`.
- Every repair action and verifier rerun is recorded in ledger evidence.
- No arbitrary shell execution is introduced.
- Full verification and `doctor` pass.
