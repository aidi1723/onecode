# OneCode v0.2 Maturity Hardening Design

Date: 2026-05-31
Status: Draft spec for the next phase after OneCode Agent Shell v0.1

## Objective

Move OneCode from a strong local Agent Kernel prototype toward a community-grade Agent platform baseline.

This phase is not about adding more flashy shell features. It is about closing the hard engineering gaps identified in the v0.1 maturity assessment:

- real sandbox boundary
- CI and release governance
- repeatable Agent benchmark
- trace-grade observability
- human approval evidence flow

## Non-Goals

- Do not integrate any gateway product.
- Do not depend on `oneword-agent-gateway`.
- Do not replace LibreChat.
- Do not add broad arbitrary shell execution.
- Do not claim production-grade maturity until sandbox, benchmark, and governance are in place.

## Architecture

v0.2 introduces five explicit layers around the existing kernel:

```text
OneCode kernel
  -> Sandbox runner
  -> Benchmark harness
  -> Trace event model
  -> Human approval model
  -> CI/release governance
```

The existing checkpoint/ledger system remains authoritative for durable run evidence. The new trace layer should reference the same `run_id`, `turn_number`, and checkpoint paths instead of creating a competing evidence system.

## P0 Sandbox Requirements

The first implementation target is Docker-based local sandboxing:

- workspace mounted read/write at a fixed in-container path such as `/workspace`
- no host filesystem access outside the mount
- sanitized environment allowlist
- optional network disablement
- CPU and memory limit fields
- process timeout
- deterministic cleanup
- explicit fallback when Docker is unavailable

The sandbox must be a kernel service, not a LibreChat frontend feature.

## P1 Benchmark Requirements

Add a local benchmark harness with fixed JSON task definitions:

- task id
- prompt
- workspace fixture
- expected status
- expected file assertions
- expected evidence assertions
- expected refusal assertions

The first benchmark set should contain at least 20 tasks before v0.2 closure.

## P1 Governance Requirements

Add minimum open-source release hygiene:

- LICENSE
- SECURITY.md
- CONTRIBUTING.md
- CI workflow for `bash scripts/verify.sh`
- release checklist
- generated-file cleanup guidance

## P2 Observability Requirements

Add a trace event model that records:

- `run_started`
- `model_call_started`
- `model_call_completed`
- `tool_call_started`
- `tool_call_completed`
- `verifier_started`
- `verifier_completed`
- `checkpoint_written`
- `human_decision_recorded`
- `run_completed`

Each event should include:

- `trace_id`
- `run_id`
- `span_id`
- `parent_span_id`
- `timestamp`
- `duration_ms` when applicable
- `status`
- compact payload

## P2 Human Approval Requirements

Introduce approval as a durable state, not just UI text:

- risky operation creates `pending_approval`
- user can approve, reject, or edit
- decision is written into evidence
- resume uses the recorded decision
- shell can display pending and completed approvals

## Closure Criteria

v0.2 is closed only when:

- sandbox smoke tests pass locally
- benchmark harness runs at least 20 tasks and writes a report
- CI workflow exists and runs the verification script
- governance docs exist
- trace events are emitted for at least model, tool, verifier, checkpoint, and run lifecycle
- approval decisions can be persisted and inspected
- full `bash scripts/verify.sh` passes

## Product Label After Closure

If v0.2 closes successfully, the project can be described as:

```text
OneCode is a local-first, sandboxed, benchmarked, evidence-preserving Agent Kernel with a LibreChat shell.
```

It should still avoid claiming broad production readiness until deployment, multi-user auth, and operational hardening are complete.
