# OneCode v0.7 Task Resume Classification Design

## Goal

Add task-level resume classification for `run-plan --resume-from`. The feature records why each planned asset or verifier is ready, still needs work, needs verification, must halt, or exposes a rule gap.

This slice extends the controlled verifier work. It does not change the existing asset-level skip behavior. It makes resume evidence explicit at task level so repeated runs can be inspected and reasoned about from ledger and manifest evidence.

## Scope

In scope:

- `run-plan` only.
- Planned write assets loaded by `load_task_plan`.
- Selected controlled verifiers from `--verifier-policy` and `--verifier`.
- Resume source evidence under `.onecode/runs/<resume-from>/manifest.json` and `ledger.json`.
- Ledger and inspect fields for task resume decisions.

Out of scope:

- `run`, `run-model`, TUI, and execution-plan resume.
- Model repair prompts.
- Changing physical write semantics.
- Reclassifying historical runs without a new resume attempt.

## Classification Model

Create `src/onecode/kernel/task_resume.py` with small dataclasses:

- `TaskResumeDecision`: `kind`, `target_type`, `target_id`, `reason`, `status_code`, `transition_action`, `transition_reason`
- `TaskResumeSummary`: list of decisions plus aggregate status fields

Decision kinds:

- `ready`: evidence proves the target already completed.
- `apply`: evidence proves the target still needs execution.
- `verify`: physical asset evidence is present but verifier evidence is missing.
- `halt`: evidence conflicts with expected state or is corrupt.
- `discover`: evidence exists but cannot be mapped to a known task rule.

Target types:

- `asset`
- `verifier`
- `task`

## Asset Decisions

For each planned asset:

- `ready`: resume manifest has a completed checkpoint for the asset path, the checkpoint payload has a SHA256, the physical file exists, and physical SHA256 matches.
- `apply`: no completed matching checkpoint exists, or the file is missing.
- `verify`: the asset is physically ready, but selected verifier evidence is missing from the source ledger.
- `halt`: source manifest or ledger is corrupt, checkpoint hash validation fails, physical file hash conflicts with checkpoint SHA256, or the checkpoint path escapes workspace.
- `discover`: source evidence has an unsupported shape that is not corrupt but cannot be classified.

`verify` is only used when controlled verifiers are selected for the resumed run. Without selected verifiers, a physically ready asset remains `ready`.

## Verifier Decisions

For each selected verifier:

- `ready`: source ledger has a matching verifier result with `status=passed`, matching id, command, cwd, and timeout.
- `apply`: no matching verifier result exists.
- `halt`: source ledger has a matching verifier id but command/cwd/timeout does not match current policy, or the result failed.
- `discover`: verifier evidence exists but lacks fields needed to compare against policy.

Verifier decisions do not execute verifiers by themselves. They tell the task whether verification is already proven or still required.

## Aggregate Task Resume Status

Aggregate decisions through `IchingKernel`:

- Any `halt` decision maps to halted resume status.
- Any `discover` decision maps to discover/inspect status.
- If all assets and verifiers are `ready`, task resume is stable and deliverable.
- If any `apply` or `verify` exists, task resume remains resumable and execution may continue through existing `run_task` and controlled verifier flow.

Add ledger fields:

- `task_resume_decisions`
- `task_resume_status_code`
- `task_resume_transition_action`
- `task_resume_transition_reason`
- `task_resume_dispatch_decision`

`inspect` should surface these fields when present.

## Execution Flow

When `run-plan --resume-from SOURCE` is used:

1. Load the task plan.
2. Load and validate verifier policy if verifiers are selected.
3. Classify planned assets and selected verifiers against the source run evidence.
4. If any decision is `halt`, stop before writes and write a halted ledger for the new run with task resume evidence.
5. Otherwise run the existing asset resume flow unchanged.
6. Run selected verifiers when needed by existing controlled verifier flow.
7. Write task resume decisions into the final ledger.

This keeps physical mutation behind existing `PathGuard`, `LogosGate`, and `run_task` behavior.

## Error Handling

- Missing source manifest: classify all assets as `apply`; this preserves existing non-fatal resume behavior.
- Missing source ledger: classify verifier targets as `apply`; assets can still be ready from manifest and physical hashes.
- Corrupt JSON in manifest or ledger: classify task as `halt` before writes.
- Physical hash conflict: classify the affected asset as `halt` before writes.
- Unsupported but parseable evidence: classify as `discover` and stop before writes.

## Testing

Implementation must be test-first. Required tests:

- A resumed verified run records asset and verifier `ready` decisions and skips ready assets.
- Missing verifier evidence records `verify` for ready assets and `apply` for verifier.
- Missing asset evidence records `apply` and writes the missing asset normally.
- Physical hash conflict records `halt` and does not write planned assets.
- Corrupt source manifest records `halt` and does not write planned assets.
- `inspect` reports task resume fields.
- Existing asset-level resume tests remain green.

## Acceptance Criteria

- `run-plan --resume-from` produces inspectable task resume decisions.
- Task resume decisions are derived from manifest, ledger, verifier policy, and physical hashes.
- Halt conditions stop before physical writes.
- Existing resume skip semantics remain unchanged.
- Full verification and `doctor` pass.
