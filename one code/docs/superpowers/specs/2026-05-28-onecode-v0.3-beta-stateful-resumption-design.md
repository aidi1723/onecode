# OneCode v0.3-beta Stateful Resumption Design

Date: 2026-05-28
Project root: `/Users/aidi/大字典/one code`
Status: Approved for implementation planning

## Goal

Add deterministic stateful resumption so a new run can import valid completed assets from a named previous run and skip already-ready work. This separates upstream API failure from local asset quality: if a previous run physically landed an asset with a matching SHA256, a retry must not repeat that write.

## Scope

v0.3-beta implements explicit resumption only:

```bash
python3 -m onecode.cli run "continue" \
  --workspace . \
  --run-id retry-002 \
  --resume-from failed-001
```

The new run reads:

```text
.onecode/runs/failed-001/manifest.json
```

It does not scan all runs automatically and does not infer user intent from latest timestamps.

### ResumeState

`resumption.py` defines:

```python
@dataclass(frozen=True)
class ReadyAsset:
    path: str
    sha256: str
    source_run_id: str
    source_turn_index: int


@dataclass(frozen=True)
class ResumeState:
    source_run_id: str
    ready_assets: dict[str, ReadyAsset]
```

`path` is normalized to the workspace-relative path used by the requested `write_text` intent. Absolute payload paths from older ledger entries must be converted back into workspace-relative paths before indexing.

### Six Ready-Asset Rules

A previous checkpoint can become a ready asset only if all conditions hold:

1. checkpoint record `status` is `completed`
2. checkpoint record `intent_type` is `write_text`
3. checkpoint record `decision` is `allowed`
4. checkpoint payload contains `path` and `sha256`
5. file still exists on disk
6. current file SHA256 equals recorded SHA256

Any failed condition means the asset is ignored. Ignored assets are non-fatal and do not stop resumption.

### Manifest Loading

`load_resume_state(workspace_root, resume_from_run_id)`:

- reads `.onecode/runs/<resume_from_run_id>/manifest.json`
- reads each checkpoint file referenced by the manifest
- validates each checkpoint against the six ready-asset rules
- returns `ResumeState(source_run_id, ready_assets)`

If the manifest does not exist, return an empty `ResumeState` with that source run id. Missing resume manifests are non-fatal in v0.3 because retry runs must still be able to write missing assets normally.

### Context

`OneCodeContext` gains:

- `resume_from_run_id: str | None`
- `resume_state: ResumeState | None`

Context creation loads the resume state only when `resume_from_run_id` is provided.

### Skip Rule

If a new run requests `write_text` for a path that is present in `resume_state.ready_assets`, the runner must:

- skip `PathGuard.write_text()`
- write a checkpoint with `status: skipped`
- set `reason: resumed_asset_ready`
- set checkpoint `intent_type: write_text`
- set checkpoint `decision: allowed`
- include payload fields: `path`, `sha256`, `source_run_id`, `source_turn_index`
- write ledger field `resumed: true`

If the requested path is not present in the resume state, the normal v0.2 `write_text` path runs.

### Manifest And Ledger

`write_checkpoint()` must support:

- `status: skipped`
- `resumed_from`
- `ready_assets` summary when available

The manifest remains logically append-only. A skipped checkpoint is still a checkpoint because it records a deterministic decision not to repeat a side effect.

### CLI

`onecode run` gains:

```bash
--resume-from <run_id>
```

The CLI passes this to `run_task()`.

## Data Flow

```text
new run input
  -> create context
  -> if --resume-from: load previous manifest
  -> build ActionIntent
  -> LogosGate.preflight
  -> if write_text path is ready: skip physical write
  -> else execute normal v0.2 path
  -> write checkpoint
  -> append manifest
  -> write ledger
```

## Testing

v0.3-beta must prove:

- CLI accepts `--resume-from`
- `load_resume_state()` reads a named previous manifest
- only valid successful `write_text` checkpoints become ready assets
- failed, denied, halted, missing-payload, missing-file, or SHA-mismatched checkpoints are ignored
- a resumed ready asset is skipped without rewriting the file
- a missing asset after resume is written normally
- timeout behavior remains `status: halted`, `partial: true`, `reason: http_timeout`
- sovereignty breach behavior remains `status: halted`, `reason: sovereignty_breach`

## Acceptance Criteria

The milestone is complete when:

- A first run writes `src/a.py`.
- A second run with `--resume-from <first-run>` and the same `--write-path src/a.py` returns `status: skipped`, `reason: resumed_asset_ready`, and `resumed: true`.
- The skipped run does not modify the file content or SHA256.
- A second run with `--resume-from <first-run>` and a different missing `--write-path src/b.py` writes normally.
- Invalid prior checkpoints are ignored, not fatal.
- The full test suite passes.

## Non-Goals

v0.3-beta does not implement:

- automatic latest-run scanning
- multi-asset planning
- model/API calls
- retry backoff
- network request replay
- conflict resolution for changed ready files
- generalized dependency graph scheduling
