# OneCode v0.7 Verifier Policy Init Design

## Goal

Make local `run-plan --verifier` setup faster by adding a CLI command that writes a safe, loadable verifier policy file for common local checks.

This is a local-development feature. It does not publish code, push branches, create PRs, or execute generated verifier commands during initialization.

## Scope

In scope:

- A new CLI subcommand for writing verifier policy JSON.
- Built-in presets for Python unittest and Python compileall.
- Workspace-bound output paths.
- Refuse overwrite by default, with explicit `--force`.
- README documentation for the local workflow.

Out of scope:

- Running verifiers during initialization.
- Project autodetection.
- Arbitrary command templates from CLI arguments.
- User-level/global config.
- TUI integration.

## CLI Surface

Add:

```bash
onecode init-verifier-policy \
  --workspace . \
  --output .onecode/verifier-policy.json \
  --preset python-unittest
```

Arguments:

- `--workspace`: defaults to `.`
- `--output`: defaults to `.onecode/verifier-policy.json`, resolved inside the workspace
- `--preset`: repeatable; defaults to `python-unittest` when omitted
- `--force`: allows overwriting an existing output file

Supported presets:

- `python-unittest`: `[sys.executable, "-m", "unittest", "discover", "-s", "tests"]`
- `python-compileall`: `[sys.executable, "-m", "compileall", "src", "tests"]`

Each preset uses `cwd: "."` and `timeout_ms: 30000`.

## Safety Rules

- The output path must resolve inside the workspace.
- Existing files are not overwritten unless `--force` is set.
- The command only writes JSON. It does not run verifier commands.
- Generated JSON must pass the existing `load_verifier_policy()` validation path.

## Output

The command prints JSON:

```json
{
  "status": "completed",
  "path": ".onecode/verifier-policy.json",
  "verifier_ids": ["python-unittest"]
}
```

On parser errors, no policy file is written.

## Testing

Required tests:

- default init writes `.onecode/verifier-policy.json` and it loads through `load_verifier_policy()`.
- repeated `--preset` writes multiple verifier specs.
- existing file is rejected without `--force`.
- `--force` overwrites an existing policy file.
- output outside workspace is rejected before writing.
- README documents the init and run-plan verifier workflow.

## Acceptance Criteria

- A local developer can generate a verifier policy without hand-writing JSON.
- Generated policy works with the existing controlled verifier loader.
- Initialization does not execute commands or broaden the safety model.
