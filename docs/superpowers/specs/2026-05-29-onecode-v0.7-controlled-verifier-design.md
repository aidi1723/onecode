# OneCode v0.7 Controlled Verifier Design

## Goal

Implement the first v0.7 task-reliability slice: policy-driven verifier execution with durable evidence. The verifier proves task completion without opening arbitrary shell execution and without bypassing `IchingKernel`.

This slice is intentionally smaller than full v0.7. It adds verifier policy, verifier result evidence, task completion gating, and inspect visibility. It does not add model-driven repair prompts, verifier presets, or TUI progress views.

## Approach

Use a dedicated verifier module instead of extending `bash_execution` or `execute_pytest`.

Alternatives considered:

- Reuse denied `execute_pytest`: too narrow and risks confusing an auditable denied intent with an allowed verifier capability.
- Add generic shell execution with guardrails: too broad for v0.7 P0 and violates the current safety model.
- Add a narrow controlled verifier policy: preferred because it gives useful verification while keeping command execution exact, bounded, and evidence-backed.

## Verifier Policy

Add `src/onecode/kernel/verifier.py` with small dataclasses:

- `VerifierSpec`: `id`, `command`, `cwd`, `timeout_ms`
- `VerifierPolicy`: allowlisted verifier specs
- `VerifierResult`: `id`, `status`, `reason`, `exit_code`, `duration_ms`, `stdout_tail`, `stderr_tail`, `stdout_sha256`, `stderr_sha256`, `cwd`, `command`

Policy rules:

- The requested verifier `id` must exist in the allowlist.
- The runtime command must match the policy command exactly.
- `cwd` must resolve inside the workspace.
- `timeout_ms` must be positive.
- Captured stdout and stderr are bounded to a fixed tail.
- Full stdout and stderr hashes are recorded.
- Verifier failure never mutates assets.

Initial policy input is file-based JSON passed to CLI commands. The schema is:

```json
{
  "verifiers": [
    {
      "id": "python-unittest",
      "command": ["python3", "-m", "unittest", "discover", "-s", "tests"],
      "cwd": ".",
      "timeout_ms": 120000
    }
  ]
}
```

## Execution Flow

Add optional verifier support to `run-plan` first. After guarded asset writes complete:

1. Load verifier policy from `--verifier-policy`.
2. Run selected verifiers from `--verifier`.
3. Convert verifier outcomes into status evidence through `IchingKernel.classify_outcome()`.
4. Compute task-level status from asset and verifier status codes.
5. Write verifier results and task completion evidence into the ledger.

The verifier command is run only after the asset phase has no failed asset. If assets halt or deny, verifier execution is skipped and the task remains non-deliverable.

## Task Evidence

Add these fields to the final run result and ledger:

- `verifier_results`
- `task_status_code`
- `task_transition_action`
- `task_transition_reason`
- `task_dispatch_decision`
- `task_entropy`
- `task_entropy_decision`
- `task_entropy_reason`
- `task_completion_evidence`
- `delivery_status`

`task_completion_evidence` records:

- `assets_complete`
- `verifiers_passed`
- `ledger_present`
- `ledger_history_present`
- `manifest_present`
- `checkpoint_count`

`delivery_status` is `deliverable` only when assets are complete, all selected verifiers pass, and evidence files are present. Otherwise it is `blocked` or `partial` according to the existing rule surface.

## CLI Surface

Extend `run-plan` with:

- `--verifier-policy PATH`
- repeatable `--verifier ID`

This keeps the first slice attached to structured task execution instead of ad hoc single writes. `run`, `run-model`, and TUI integration remain out of scope.

## Inspect Output

`onecode inspect` should surface verifier and task-level fields when present. Existing runs without these fields remain valid.

## Error Handling

- Missing or invalid policy file: CLI parser error before writes.
- Unknown verifier id: parser error before writes.
- Policy command mismatch: parser error before writes.
- CWD outside workspace: parser error before writes.
- Verifier non-zero exit: run result is not deliverable; evidence records exit code and output hashes.
- Verifier timeout: run result is not deliverable; evidence records timeout reason and bounded output.

## Testing

Implementation must be test-first. Required tests:

- `run-plan` with a passing verifier records `verifier_results` and `delivery_status=deliverable`.
- verifier failure blocks delivery and records exit code, tails, and hashes.
- verifier timeout records `reason=verifier_timeout`.
- unknown verifier id fails before asset writes.
- verifier cwd outside workspace fails before asset writes.
- denied `bash_execution` and `execute_pytest` remain denied.
- `inspect` reports task-level verifier fields.

## Acceptance Criteria

- A structured task can write assets, run an allowlisted verifier, and inspect the evidence.
- No arbitrary shell capability is exposed.
- All task-level decisions are derived from recorded asset and verifier evidence.
- Existing v0.6 tests remain green.
