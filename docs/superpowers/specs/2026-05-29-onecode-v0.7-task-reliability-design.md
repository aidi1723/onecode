# OneCode v0.7 Task Reliability Design

## Goal

OneCode v0.7 turns the v0.6 rule-closed execution kernel into a task reliability layer: an agent can execute repeatable work, prove what happened, recover from partial progress, and reach an auditable completion decision.

The goal is not to build another coding assistant. The goal is to make agent work controllable, repeatable, recoverable, and evidence-backed.

## Core Rule

Task reliability is the product goal. The Iching rule surface is the only control plane.

Every new task execution feature must answer these questions before entering the kernel:

- What evidence does it sample?
- How does that evidence collapse into Liangyi, Sixiang, Bagua, and the 64-hexagram status space?
- How does yin-yang pressure regulate it?
- How do five-element relations modulate it?
- What does `IchingKernel.transition()` decide?
- What checkpoint, manifest, ledger, and trace evidence proves it?
- How does resume reproduce the same decision without guessing?

No v0.7 feature may add a parallel control variable that bypasses `IchingKernel`.

## Non-Goals

v0.7 does not:

- remove Iching, yin-yang, five-element, or 64-hexagram semantics
- expose arbitrary shell execution
- become a general coding assistant
- optimize for free-form model autonomy over deterministic completion
- treat verifier output as authority outside the rule surface

## Reliability Model

OneCode v0.7 introduces task-level reliability above asset-level writes.

The existing asset-level model remains:

- `write_text` and `patch_text` are guarded physical mutations
- `PathGuard` owns path sovereignty
- `LogosGate` owns bounded execution and preflight
- `checkpoint`, `manifest`, `ledger.json`, and `ledger.jsonl` own evidence
- `IchingKernel` owns status and transition semantics

v0.7 adds task-level closure:

- `task_status_code`
- `task_transition`
- `task_dispatch_decision`
- `task_entropy`
- `task_completion_evidence`
- `task_resume_decision`

Task state is not a new policy engine. It is the aggregate projection of step, asset, verifier, and resume evidence through `IchingKernel`.

## Lifecycle Mapping

The task lifecycle maps onto the existing rule stack:

| Task Reliability Concept | Rule Projection |
| --- | --- |
| raw user objective and runtime telemetry | Taiyi evidence space |
| executable / blocked / observed facts | Liangyi bits |
| read, write, skip, conflict phases | Sixiang windows |
| environment and asset planes | Bagua trigrams |
| whole task control state | 64-hexagram status |
| repeated success or repeated failure | yin-yang polarity |
| verifier, write, network, sandbox, and sovereignty interaction | five-element modulation |
| resumable interruption | recovery-seed checkpoint |
| unsafe boundary | sovereignty fire halt |
| final stable completion | yang cooldown into deliverable state |

## Task-Level Status

Every task run must produce a task-level status envelope:

```json
{
  "task_status_code": 63,
  "task_transition": {
    "status_code": 39,
    "action": "cooldown",
    "reason": "yang_overload_cooldown"
  },
  "task_dispatch_decision": "continue",
  "task_entropy": 0.0,
  "task_entropy_decision": "accept_positive_polarity",
  "task_entropy_reason": null
}
```

The task-level status is derived from:

- step results
- asset results
- verifier results
- resume audit events
- delivery inspection

It must not be manually assigned by the model.

## Completion Evidence

A task is complete only when all required completion evidence is present.

Minimum completion evidence:

- all required assets are completed or intentionally skipped through resume
- all required patch evidence includes pre/post and block hashes
- all required verifier checks pass
- no unresolved halt, sovereignty breach, invalid intent, or failed dependency remains
- manifest checkpoint hashes match physical checkpoint files
- ledger snapshot and ledger history are present
- final task transition is deliverable or stable

If any required evidence is missing, the task cannot report final completion. It must report partial, blocked, or resumable state through the rule surface.

## Controlled Verifier

v0.7 may add verifier execution, but it must not open arbitrary shell execution.

Verifier execution must be policy-driven:

```json
{
  "verifiers": [
    {
      "id": "python-unittest",
      "command": ["python", "-m", "unittest", "discover", "-s", "tests"],
      "cwd": ".",
      "timeout_ms": 120000
    }
  ]
}
```

Verifier constraints:

- command must match a policy allowlist entry exactly
- cwd must resolve inside the workspace
- timeout is mandatory
- stdout and stderr are captured with bounded tails
- exit code, duration, and output hashes enter evidence
- verifier result maps to `IchingKernel.classify_outcome()` or a v0.7 classifier extension
- verifier failure cannot directly mutate assets

`bash_execution` and `execute_pytest` remain denied intent types until this controlled verifier model exists.

## Resume Semantics

Task resume upgrades the current asset resume model.

Resume must classify every step as one of:

- `ready`: evidence proves the step is already complete
- `apply`: evidence proves the step must still run
- `verify`: physical output exists but task-level verifier evidence is missing
- `halt`: evidence conflicts with expected state
- `discover`: the evidence does not map to a known task rule

These classifications must be converted into status codes and transitions through `IchingKernel`. Resume must not rely on ad hoc booleans outside the rule surface.

## Repair Loop

v0.7 repair is evidence-driven:

- failed verifier output becomes evidence, not instruction authority
- repair prompts include failed step, failed verifier, affected paths, reasons, and status profile
- repair plans are patch-only unless policy explicitly allows broader action
- each repair attempt writes checkpoint evidence
- `max_repair_attempts` is honored deterministically
- repeated failed repair must halt or discover through the rule surface

## Delivery Decision

Final delivery must be task-level, not merely file-level.

Delivery states:

- `deliverable`: assets complete, verifiers pass, evidence chain complete
- `partial`: some required work remains but resume is possible
- `blocked`: halt or conflict requires external decision
- `discover`: rule gap requires new mapping

`inspect` and `audit-self` should display the delivery state with the relevant task status code, transition, entropy decision, and evidence summary.

## Evidence Additions

v0.7 should add these evidence fields:

- `task_status_code`
- `task_transition_action`
- `task_transition_reason`
- `task_dispatch_decision`
- `task_entropy`
- `task_entropy_decision`
- `task_entropy_reason`
- `task_completion_evidence`
- `verifier_results`
- `task_resume_decisions`
- `delivery_status`

These fields should appear consistently in execution traces, ledger results, manifest summaries, and inspect output.

## Test Strategy

v0.7 implementation must be test-first.

Required test classes:

- task completes only when assets and verifiers pass
- completed task rerun is idempotent and skips ready work
- verifier failure blocks delivery and produces evidence
- verifier timeout maps to recovery or halt through `IchingKernel`
- task resume distinguishes ready/apply/verify/halt/discover
- repair loop uses verifier failure evidence
- `bash_execution` and `execute_pytest` remain denied outside controlled verifier policy
- inspect reports task-level delivery state
- no forbidden parallel control variables are introduced

## Implementation Priority

P0:

- task-level status envelope
- controlled verifier policy and result evidence
- task completion evidence gate
- task-level resume classification

P1:

- verifier failure repair integration
- inspect/report task delivery summary
- idempotent repeated task execution tests

P2:

- reusable policy schema
- verifier presets for Python and Node projects
- task templates

P3:

- richer TUI presentation
- long-running task progress views
- external agent adapters

## Acceptance Criteria

v0.7 is acceptable when:

- a task can be run, verified, resumed, and inspected from evidence alone
- repeated runs produce deterministic skip/apply/halt decisions
- verifier execution is useful without allowing arbitrary shell access
- every task-level decision has a status code, transition, and evidence record
- full verification and `onecode audit-self` pass
- no task reliability feature bypasses `IchingKernel`
