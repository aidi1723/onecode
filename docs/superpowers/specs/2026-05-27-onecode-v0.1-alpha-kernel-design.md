# OneCode v0.1-alpha Kernel Design

Date: 2026-05-27
Project root: `<private-development-workspace>`
Status: Approved for implementation planning

## Goal

Build the first clean, independent OneCode kernel. The v0.1-alpha milestone proves deterministic state control, checkpointed persistence, source-hygiene discipline, and timeout-bound halt behavior before any TUI, benchmark suite, model adapter, or multi-agent planner is added.

## Boundary

OneCode uses a dual-track structure:

- `<local-user-path>` remains the historical validation and benchmark repository.
- `<private-development-workspace>` is the new clean OneCode product root.

Historical benchmark results, docs, and concepts can inform requirements, but code is not automatically migrated. Every OneCode source file must be independently authored inside the new project root.

## Source Hygiene

OneCode v0.1-alpha follows these source rules:

- Public clean-room or permissively licensed open-source agent projects may be studied only for general architecture, after license review.
- Minimal educational rewrites may be studied only for broad concepts such as command loops, memory scopes, and UI interaction categories.
- Leaked, mirrored, or DMCA-risk Claude Code source repositories are forbidden as code sources, naming sources, control-flow sources, or file-structure sources.
- The historical one-word gateway repository is reference evidence only. Implementation must be new unless a later migration plan explicitly identifies a file, license status, reason, and rewrite strategy.

## Architecture

The first kernel is a small Python package with a CLI smoke path. It has no real model calls and no terminal UI. It runs a minimal deterministic turn, writes physical evidence, and exits with a manifest-backed result.

Planned modules:

```text
src/onecode/
  __init__.py
  cli.py
  kernel/
    __init__.py
    checkpoint.py
    context.py
    hexagram.py
    logos_gate.py
    runner.py
```

### Hexagram Status

`HexagramStatusCode` represents a strict six-bit state code. v0.1-alpha treats the code as a typed string and validates that every value is exactly six binary digits.

Initial semantic states:

| Code | Meaning |
| --- | --- |
| `111111` | build/action entry |
| `000001` | verify/checkpoint gate |
| `110000` | correction/feedback gate |
| `101000` | inspect/read gate |
| `000000` | complete or halted ground state |

The initial implementation does not claim a complete 64-state ontology. It only enforces the mechanical representation and records transitions.

### Context

`OneCodeContext` contains:

- `run_id`
- `workspace_root`
- `evidence_root`
- `state`
- `turn_index`
- `manifest_path`
- `http_timeout_seconds`

The context is deliberately small. It does not store rolling conversation history. Durable state lives in checkpoint and manifest files.

### LogosGate

`LogosGate` owns hard runtime permission decisions for v0.1-alpha:

- validates timeout settings
- runs bounded actions
- converts timeout into a deterministic halt result
- marks timeout exits as `partial: true`
- forces the next state to `000000` after a hard halt

The default network/action timeout is 60 seconds. The CLI exposes it as:

```bash
onecode run "task" --http-timeout-seconds 60
```

### Checkpoint And Manifest

Every completed turn writes evidence under:

```text
.onecode/runs/<run_id>/
  manifest.json
  ledger.json
  checkpoints/
    0001.json
```

Checkpoint writes are atomic:

1. Serialize checkpoint JSON.
2. Write to a temporary file in the target directory.
3. Flush the file.
4. Rename the temporary file to the final checkpoint path.
5. Update `manifest.json` with the checkpoint path and SHA256.

`manifest.json` records:

- run id
- created timestamp
- workspace root
- current state
- checkpoint records
- final status when known
- `partial` flag
- halt reason when known

`ledger.json` records the final user-facing run result.

## Data Flow

The v0.1-alpha run flow is:

```text
task input
  -> create OneCodeContext(state = 111111)
  -> LogosGate preflight
  -> execute bounded kernel action
  -> write checkpoint
  -> update manifest with checkpoint SHA256
  -> compute next state
  -> write final ledger
```

The first bounded action is intentionally simple: it records a controlled echo of the requested task and checkpoint metadata. This proves the state, persistence, timeout, and CLI surfaces without adding model or tool complexity.

## Timeout Behavior

If a bounded action exceeds `http_timeout_seconds`, the run does not retry in v0.1-alpha. It writes a partial halt result:

```json
{
  "status": "halted",
  "partial": true,
  "reason": "http_timeout",
  "next_state": "000000"
}
```

The halt path must still write both checkpoint and ledger evidence before returning.

## CLI

The initial CLI command is:

```bash
python3 -m onecode.cli run "build a small asset" --workspace . --http-timeout-seconds 60
```

It prints compact JSON containing:

- `run_id`
- `status`
- `state`
- `manifest_path`
- `ledger_path`
- `partial`
- `reason`

## Testing

v0.1-alpha requires focused automated tests for:

- valid and invalid six-bit status codes
- context creation and run directory layout
- checkpoint JSON creation
- manifest SHA256 matching the checkpoint file
- normal CLI smoke run
- timeout path producing `status: halted`, `partial: true`, and `reason: http_timeout`
- source hygiene document presence and forbidden-source rules

## Non-Goals

v0.1-alpha does not include:

- real LLM/model calls
- React + Ink TUI
- benchmark A/B harness
- external GitHub repository ingestion
- multi-agent Planner/Executor orchestration
- migration of historical gateway code
- complete 64-state policy ontology

## Acceptance Criteria

The milestone is complete when:

- `python3 -m onecode.cli run "smoke" --workspace .` exits successfully.
- The run creates `.onecode/runs/<run_id>/manifest.json`.
- The manifest contains at least one checkpoint record with a matching SHA256.
- A forced timeout run exits through the partial halt path and writes evidence.
- The test suite covers the normal and timeout paths.
- No implementation file depends on leaked-source mirrors or historical repository code.
