# 测 Opcode Workflow

## Purpose

执行测试、构建、检查和验证，输出系统层证据，不伪造通过结果。

## Prompt Engineering Sources

- Test-driven development verification pattern: 测试先证明行为，再支撑实现。
- CI quality gate pattern: 用命令退出码、stdout、stderr 和覆盖率做判断。
- Evidence-before-claims pattern: 先运行验证，再汇报状态。

## Root Skill Mount

- 挂载名：`tdd_ci_quality_gate_mount`
- 社区参考源：pytest-cov、verification-before-completion。
- 挂载动作：加载变更文件、测试命令、通过标准和覆盖率阈值。
- 工具倾向：`run_pytest`、`run_npm_test`、`run_build`、coverage report。
- 硬门：没有 stdout/stderr/exit code 不得签发通过；覆盖率不足必须如实失败。

## Efficiency Controls

- 优先运行和变更范围直接相关的最小测试集。
- 必要时再升级到全量测试、构建、类型检查或 lint。
- 复用项目已有脚本，不发明新的验证入口。

## Precision Controls

- 必须记录验证命令、退出码、关键输出和失败摘要。
- 区分测试失败、命令不可用、环境缺失和证据不足。
- 不能凭模型推理替代真实命令输出。

## Stability Controls

- 默认 Execute-Only，不修改业务代码。
- 测试失败转 `修` 或 `查`，高风险命令转 `卫`。
- 验证输出由系统沙盒捕获并生成摘要哈希。

## Six Phases

1. 确认验证对象和通过标准。
2. 选择项目已有测试或构建命令。
3. 运行命令并捕获 stdout、stderr 和 exit code。
4. 生成证据哈希。
5. 判断通过、失败或证据不足。
6. 成功转总或记，失败转修或查。
