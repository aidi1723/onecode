# 修 Opcode Workflow

## Purpose

在受限范围内修复已有代码问题，必须先复现，再做最小修改，再验证。

## Prompt Engineering Sources

- Systematic debugging pattern: 先捕获真实失败，再定位根因，不凭感觉改代码。
- Test-driven development pattern: 能补回归测试时先补测试，再做最小实现。
- Surgical change pattern: 只改受影响文件和必要逻辑，保持公开接口稳定。
- Verification-before-completion pattern: 没有系统证据不得宣称已修复。

## Root Skill Mount

- 挂载名：`surgical_fix_mount`
- 社区参考源：SWE-agent ACI、Superpowers systematic debugging / TDD。
- 挂载动作：加载失败证据、最小复现、影响范围和上一轮测试输出。
- 工具倾向：只读检索、受限编辑、撤销修改、linter。
- 硬门：没有复现不得宣称修复；禁止顺手重构；禁止覆盖用户既有逻辑。

## Efficiency Controls

- 从最小复现和最近失败日志开始，避免全仓库漫游。
- 修改前先确定影响范围，避免顺手重构。
- 多次失败后熔断到 `查`，保留证据，停止继续消耗上下文。

## Precision Controls

- 根因必须区分配置、环境、依赖、逻辑、并发、权限或数据问题。
- 变更说明必须说明改了什么、为什么改、验证命令是什么。
- 不得把“测试未运行”描述成“测试通过”。

## Stability Controls

- 写权限限制在受影响文件。
- 禁止未批准安装依赖、删除用户改动或大面积重写。
- 每次执行字切换后触发上下文清理，只保留原始需求、diff 和验证证据。

## Six Phases

1. 捕获真实失败上下文。
2. 建立最小复现路径，保留最小复现证据。
3. 定位影响范围。
4. 执行最小必要修改。
5. 运行验证命令。
6. 成功转测或审，失败熔断到查或停。

## Evidence

- reproduction_result
- diff_summary
- test_output_hash
- modified_line_numbers
