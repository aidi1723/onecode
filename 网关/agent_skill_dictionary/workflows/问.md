# 问 Opcode Workflow

## Purpose

在需求、权限、目标或上下文不足时，提出最小必要澄清问题。

## Prompt Engineering Sources

- Brainstorming / requirement clarification pattern: 先澄清目标、约束和成功标准。
- PermissionRequest pattern: 高风险权限由用户确认。
- Human-in-the-loop orchestration pattern: 低置信度时暂停自动判断。

## Root Skill Mount

- 挂载名：`human_in_the_loop_mount`
- 社区参考源：LangGraph interrupts、Permission request pattern。
- 挂载动作：加载歧义摘要、风险摘要和结构化选项 Schema。
- 工具倾向：`send_user_message`、`render_ui_options`。
- 硬门：禁止猜测用户意图；没有用户确认不得恢复高风险流程。

## Efficiency Controls

- 最多提出少量关键问题，优先问会改变执行路径的问题。
- 给出推荐选项和影响，降低用户决策成本。
- 用户回答后立刻结构化为执行字、参数和权限确认。

## Precision Controls

- 问题必须指向具体缺口，例如目标、范围、文件、权限、环境或验收标准。
- 不用开放式长问卷阻塞流程。
- 用户确认结果必须结构化保存，供后续执行字读取。

## Stability Controls

- 默认不写文件、不执行命令、不调用外部网络。
- 缺少关键权限或需求时不得猜测执行。
- 用户没有确认前，高风险流程保持挂起。

## Six Phases

1. 定位缺失信息或冲突意图。
2. 压缩成最少问题。
3. 给出选项和影响。
4. 等待用户输入。
5. 结构化确认结果。
6. 重新编译执行字或恢复执行栈。
