# 查 Opcode Workflow

## Purpose

只读调查项目、文件、结构、日志、配置和外部来源，不修改任何业务文件。

## Prompt Engineering Sources

- Reader-only agent pattern: 只负责读取、检索、定位和整理事实，不接触写权限。
- Claude / Codex skill on-demand context pattern: 只加载完成调查所需的最小上下文。
- Code review evidence pattern: 每个结论尽量绑定文件路径、行号、命令输出或来源。

## Root Skill Mount

- 挂载名：`inspect_repo_map_mount`
- 社区参考源：Aider Repository Map、SWE-agent Agent-Computer Interface。
- 挂载动作：生成压缩目录树、符号地图、目标文件和行号引用。
- 工具倾向：`rg`、`git_diff`、repo map generator、只读文件读取。
- 硬门：禁止写入、安装依赖或破坏性命令；结论必须绑定证据。

## Efficiency Controls

- 优先使用快速索引和搜索工具，例如 `rg`、文件树、清单和已有文档。
- 只读取和问题直接相关的文件片段，避免把整个仓库塞进上下文。
- 输出下一执行字建议，而不是在调查阶段顺手修改代码。

## Precision Controls

- 明确区分事实、推断和未知项。
- 文件结论必须带路径；代码结论尽量带行号；外部信息必须带来源。
- 没有证据时只说“未确认”，不得补全不存在的实现细节。

## Stability Controls

- 权限锁定为 Read-Only。
- 遇到需要写入、安装依赖、联网或执行危险命令的情况，转 `问` 或 `卫`。
- 发现上下文不足时先收敛问题，不扩大调查范围。

## Six Phases

1. 确认调查目标。
2. 定位相关文件、日志、配置或来源。
3. 读取最小必要上下文。
4. 区分事实、推断和未知项。
5. 输出证据路径、行号或来源。
6. 推荐下一执行字。
