# 停 Opcode Workflow

## Purpose

冻结执行栈，收回高风险权限，等待人工确认恢复、降级或终止。

## Prompt Engineering Sources

- Guardrail interrupt pattern: 触发安全或质量断路器时停止自动执行。
- Human-in-the-loop recovery pattern: 高风险恢复必须等待人类确认。
- Meltdown-to-readonly pattern: 多次失败后降级到只读调查，避免越改越乱。

## Root Skill Mount

- 挂载名：`circuit_breaker_mount`
- 社区参考源：Circuit Breaker Pattern、LangGraph interrupt pattern。
- 挂载动作：冻结失败计数、最后安全状态、执行栈和恢复选项。
- 工具倾向：无工具；默认不转发上游模型。
- 硬门：触发后不继续生成方案；恢复必须经过人工确认或明确安全策略。

## Efficiency Controls

- 触发后不再调用大模型继续扩写方案。
- 只保留失败证据、权限状态、当前执行栈和可选恢复路径。
- 恢复时从明确执行字重新进入，不继承污染上下文。

## Precision Controls

- 必须说明触发原因、失败次数、被收回权限和待确认动作。
- 不提供模糊的“继续试试”，只提供恢复、降级、终止三类结构化选择。
- 任何恢复都必须绑定用户确认。

## Stability Controls

- 默认 Zero-Token / No-Write。
- 写入、部署、依赖安装、外部网络和 destructive command 全部冻结。
- 只有用户确认或明确安全策略能解除冻结。

## Six Phases

1. 冻结当前执行栈。
2. 收回写入、部署、依赖安装和高风险工具权限。
3. 汇总失败证据和触发原因。
4. 给出恢复、降级或终止选项。
5. 等待用户确认。
6. 根据确认恢复到指定执行字或终止。
