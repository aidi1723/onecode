# V0.3 行动框架：八大 Opcode 原型落地

日期：2026-05-24
状态：已实施基础字段、继承校验、根字 workflow markdown、workflow loader 和网关注入
前置：V0.2 第二阶段已完成

## 1. V0.3 目标

V0.3 不扩写 64 字。V0.3 只做一件事：

```text
把 8 个根字固化为 AgentOS 的底层行为原语，并让现有 22 个执行字继承它们。
```

完成后，一字诀会从“执行字词典”升级为“带继承规则的行为原语内核”。

同时，V0.3 要明确一个关键工程原则：

```text
每个根字背后都是一套精准、可靠、稳定、专业的提示词工程和规范。
```

这套规范不只写在白皮书里，而是拆成 workflow markdown，由 `workflow_loader` 加载，并在网关 system rule 中按 `root_opcode` 注入。

## 2. 不做什么

V0.3 明确不做：

- 不一次性设计 64 个字。
- 不引入复杂卦象解释系统。
- 不把状态转移权交给大模型。
- 不重写当前网关。
- 不改变现有 `/v1/chat/completions` 兼容入口。

这次升级必须保持 V0.2 的稳定主链路。

## 3. 交付物

### 3.1 Schema 扩展

在 `schemas/agent-skill-dictionary.schema.json` 中新增字段：

```text
root_opcode
opcode_vector
inheritance_policy
six_phase_workflow
transition_policy
```

### 3.2 词典升级

在 `agent_skill_dictionary/programming-agent-skill-dictionary.json` 中：

- 给 22 个执行字补 `root_opcode`。
- 给 8 个根字补完整 `opcode_vector`。
- 给所有字补 `inheritance_policy`。
- 给 8 个根字补 `six_phase_workflow`。
- 给所有字补最小 `transition_policy`。

### 3.3 Validator 升级

在 `agent_skill_dictionary/validator.py` 中增加检查：

- 每个字必须有 `root_opcode`。
- `root_opcode` 必须是 8 个根字之一。
- 根字的 `root_opcode` 必须等于自己。
- 子字不能放宽父字权限。
- 子字必须继承或增加证据要求。
- 高风险字必须有 `on_risk` 转移到 `卫` 或 `停`。

### 3.4 网关注入升级

在 `agent_skill_dictionary/gateway_core.py` 中：

- system rule 注入当前字规则。
- 同时注入根字规则摘要。
- 同时注入根字 Workflow 摘要。
- 注入六步 workflow。
- 注入 transition policy。

### 3.5 根字 Workflow 文件

新增：

```text
agent_skill_dictionary/workflow_registry.json
agent_skill_dictionary/workflow_loader.py
agent_skill_dictionary/workflows/查.md
agent_skill_dictionary/workflows/修.md
agent_skill_dictionary/workflows/测.md
agent_skill_dictionary/workflows/卫.md
agent_skill_dictionary/workflows/停.md
agent_skill_dictionary/workflows/问.md
agent_skill_dictionary/workflows/记.md
agent_skill_dictionary/workflows/总.md
```

每个 workflow 文件必须包含：

```text
Prompt Engineering Sources
Efficiency Controls
Precision Controls
Stability Controls
Evidence
```

这些章节把社区优秀 Skill 和 Agent workflow 的精髓变成根字运行规范：

- `查` 参考 Reader-only、按需上下文和证据绑定模式。
- `修` 参考 systematic debugging、TDD、surgical change 和 verification-before-completion。
- `测` 参考 CI quality gate、真实命令输出和 evidence-before-claims。
- `卫` 参考 PreToolUse、Guardrails tripwire 和 zero-trust agent。
- `停` 参考 interrupt/resume、meltdown-to-readonly 和 human recovery。
- `问` 参考需求澄清、PermissionRequest 和 human-in-the-loop。
- `记` 参考 CLAUDE.md / AGENTS.md、ADR 和 docs-as-source-of-truth。
- `总` 参考 context compaction、handoff summary 和 evidence-preserving summary。

### 3.6 测试升级

新增或扩展测试：

- `tests/test_opcode_primitives.py`
- `tests/test_workflow_loader.py`

- 检查 `停` 是高优先级控制字。

## 6. Macro Chain 编译规则

V0.3 先不做复杂规划模型，只做确定性宏链模板。

初始模板：

| 用户意图组合 | Macro Chain |
| --- | --- |
| 查问题并修复验证 | `查 -> 修 -> 测 -> 总` |
| 安全风险修复 | `查 -> 卫 -> 修 -> 测 -> 总` |
| 需求不清的新功能 | `问 -> 查 -> 修 -> 测 -> 记 -> 总` |
| 部署前检查 | `查 -> 卫 -> 测 -> 停` |
| 文档沉淀 | `查 -> 记 -> 总` |

Macro Chain 不是让模型自由生成，而是由网关或后续 compiler 根据规则选择。

## 9. 决策记录

V0.3 采用“八字为骨，六十四为展”的架构：

```text
8 个根字约束物理边界
22 个现有字继承根字
64 字只作为未来扩展，不进入本阶段交付
```

这保证系统不会被过早的 64 字设计拖散，也保留了以后扩展成完整指令集的空间。
