# Build Mode 内核运行规则 V2

日期：2026-05-26
定位：一字诀 AgentOS 的完整项目构建执行法典
状态：设计法典；尚未作为运行时代码完全落地

## 1. 核心结论

Build Mode 解决的是上一轮真实 A/B 暴露出的关键问题：

```text
只防御会交白卷，只放权会失主权。
Build Mode 必须能写、能测、能修，但不能越权。
```

它不是继续加拦截器，而是把 Agent 的执行周期重构为一条状态驱动的最小操作系统管道：

```text
两仪判别 -> 四象分流 -> 八卦定权 -> 证据变卦 -> 归档收束
```

最高铁律：

```text
先判卦，再给权。
一卦一权，过卦收权。
失败不崩，越权必停。
证据不足，不许完成。
```

## 2. 与现有八根字的关系

当前仓库 V1 根字仍以 `查 / 修 / 测 / 卫 / 停 / 问 / 记 / 总` 为准。V2 Build Mode 引入的是更偏执行流的运行语义，不应直接打乱现有根字文件和测试。

为避免状态码冲突，V2 采用“兼容演进”：

| V2 执行卦 | Bit | V2 语义 | V1 兼容根字 | 说明 |
| --- | --- | --- | --- | --- |
| 坤 | `000` | 归档、只读固化 | `记` | 负责 manifest、hash、归档记录；最终交接仍可进入 `总` |
| 震 | `001` | 沙盒测试 | `测` | V2 将测试执行明确绑定瞬时隔离环境 |
| 坎 | `010` | 风险隔离、影子 I/O | `卫` | 作为安全守卫的隔离派生形态 |
| 巽 | `011` | 零工具纯问答 | `问` | 纯文本快轨，`tools=[]`，不写盘、不执行 |
| 艮 | `100` | 主权熔断 | `停` | 越权路径、危险命令、宿主机破坏直接停止 |
| 离 | `101` | repo card、上下文注入 | `查` | 只读资产拓扑与错误上下文脱水 |
| 兑 | `110` | 纠错、软反馈 | `问` | 将失败、拒绝、退出码转为可继续的结构化反馈 |
| 乾 | `111` | 造物、scoped 写入 | `修` / 派生 `造` | 允许工作区内创建和修改文件，但不允许越权 |

工程解释：

- `造` 仍然是 `修` 的派生字，不直接替换 V1 根字。
- `纠` 是 `问` 的派生状态，负责失败反馈而不是用户澄清。
- `归` 是 `记` 的派生状态，负责最终物理资产固化。
- `隔` 是 `卫` 的派生状态，负责风险缓冲和影子 I/O。
- `总` 在 V1 中仍表示最终摘要、交接和上下文压缩；V2 中的 `坤/归` 处理物理归档后，可以再进入 `总` 输出人类可读报告。

## 3. 第一层：两仪判别

任何请求进入 Build Mode 前，先做二元判别：

```text
阳 = 1 = 需要改变外部环境
阴 = 0 = 不应改变外部环境，或必须收敛/防御
```

判为阳的典型信号：

- 创建文件、目录或项目。
- 修改源码、配置、测试或文档。
- 安装依赖。
- 运行测试、构建、格式化、迁移脚本。
- 需要生成可落盘资产。

判为阴的典型信号：

- 纯解释、纯咨询、只读审查。
- 安全扫描、越权拦截、风险隔离。
- 最终归档、哈希、上下文压缩。
- 低置信度澄清。

注意：`运行测试` 从用户意图看属于阳性动作，因为它会派生进程；但在执行控制上必须进入阴性隔离空间。

## 4. 第二层：四象分流

两仪之后，系统把动作投影到四个物理作用域：

| 四象 | Bit | 空间 | 用途 | 默认工具策略 |
| --- | --- | --- | --- | --- |
| 太阳 | `11` | 特权写入空间 | scoped 写盘、创建项目、应用补丁 | 只开放写入类工具 |
| 少阴 | `10` | 虚拟隔离空间 | 容器、临时 venv、测试执行 | 只开放验证类工具 |
| 少阳 | `01` | 纯净推理空间 | 轻量问答、错误理解、上下文消化 | `tools=[]` 或只读卡片 |
| 太阴 | `00` | 绝对物理边界 | PATH 哨兵、越权熔断、归档锁定 | 默认无工具 |

四象不是对用户可见的提示词，而是系统内部决定“动作在哪里发生”的物理路由。

## 5. 第三层：八卦运行矩阵

每个时间片只能存在一个 active primitive。当前 primitive 决定工具、路径、上下文、证据和失败转移。

| 卦 | Bit | 原语 | 允许动作 | 禁止动作 | 必须证据 | 失败转移 |
| --- | --- | --- | --- | --- | --- | --- |
| 坤 | `000` | 归 | 生成 manifest、SHA256、只读固化、清理临时资源 | 新增业务改动、继续跑命令 | `manifest_path`、`sha256_map`、`readonly_status` | `停` |
| 震 | `001` | 测 | 在隔离环境内运行 `pytest`、构建、检查 | 宿主机裸跑、写业务源码 | `exit_code`、`stdout_digest`、`stderr_digest`、`duration_ms` | `兑` 或 `停` |
| 坎 | `010` | 隔 | 影子文件系统、RAM buffer、风险缓冲 | 真实写盘、真实外联 | `shadow_fs_id`、`blocked_io`、`risk_reason` | `兑` 或 `停` |
| 巽 | `011` | 问 | 纯文本推理、澄清、轻量咨询 | 工具调用、写盘、执行 | `answer_text` 或 `clarification_request` | `问` 或 `停` |
| 艮 | `100` | 停 | 拒绝越权、冻结现场、返回恢复条件 | 继续转发高风险动作 | `block_reason`、`blocked_action`、`recovery_condition` | 人类确认或 `离` |
| 离 | `101` | 查 | 只读 repo card、错误定位、上下文注入 | 直接编辑、安装依赖、运行危险命令 | `repo_card`、`target_files`、`line_refs` | `乾`、`巽` 或 `停` |
| 兑 | `110` | 纠 | 压缩失败信息、软反馈、保持连接 | 原样倾倒海量日志、伪造通过 | `feedback_json`、`next_recommended_state` | `离` |
| 乾 | `111` | 造 | scoped `make_dir`、`write_file`、`patch` | 工作区外写入、shell 绕路、读写密钥目录 | `changed_files`、`path_scope`、`patch_digest` | `震` 或 `艮` |

## 6. Build Mode 标准流转

完整项目构建默认使用以下闭环：

```text
乾(造) -> 震(测) -> 兑(纠) -> 离(查) -> 乾(修/造) -> 震(测) -> 坤(归) -> 总
```

### 6.1 正常路径

1. `intent_resolver` 识别项目构建、功能开发或修复任务。
2. 进入 `乾 111`，只开放 scoped 写入工具。
3. 写入完成后进入 `震 001`，只开放沙盒测试工具。
4. 测试通过进入 `坤 000`，生成 manifest、hash、只读归档。
5. 归档后进入 V1 `总`，输出交付摘要、剩余风险和复现命令。

### 6.2 测试失败路径

```text
震 001 -> 兑 110 -> 离 101 -> 乾 111 -> 震 001
```

规则：

- `震` 不把完整日志塞回模型。
- `兑` 将失败压缩成短 JSON。
- `离` 注入最小上下文卡，只包含失败文件、行号、断言和相关片段。
- `乾` 重新获得 scoped 写入权，做最小修复。
- 连续失败超过阈值转 `停`。

### 6.3 越权路径

```text
任意状态 -> 艮 100 -> 兑 110 -> 离 101 -> 原合法状态或停
```

越权包括：

- 路径逃逸：`../`、绝对路径写入、敏感目录读写。
- 高危命令：`rm -rf`、`curl | sh`、任意 shell 绕过。
- 密钥目录：`~/.ssh`、`~/.codex`、`~/.claude`、`.env` 非授权读取。
- 宿主机破坏：修改系统配置、杀无关进程、改全局 PATH。

`艮` 的阻断结果不应导致客户端崩溃。由 `兑` 转为结构化软反馈，再由 `离` 注入边界原因。

## 7. MVP 六大模块

### 7.1 `intent_resolver`

职责：

- 判断阴阳。
- 映射四象。
- 计算初始八卦。
- 输出置信度和原因。

接口草案：

```python
IntentDecision(
    yin_yang="yang",
    quadrant="11",
    primitive="111",
    label="造",
    confidence=0.91,
    reasons=["project_creation", "requires_file_write", "requires_tests"],
)
```

### 7.2 `permission_matrix`

职责：

- 根据 active primitive 生成工具白名单。
- 过滤请求中的工具 Schema。
- 设置输出预算、温度、上下文模式。
- 生成“当前世界线”系统规则。

硬规则：

```text
模型在任意时间只能看到当前卦允许的工具。
```

### 7.3 `scoped_writer`

职责：

- 执行 `乾` 的写盘动作。
- 将所有相对路径解析到工作区根目录。
- 拒绝路径逃逸和敏感目录。
- 输出 changed files 和 patch digest。

路径规则：

```text
realpath(target).startswith(realpath(workspace_root)) 必须为真。
```

### 7.4 `sandbox_runner`

职责：

- 执行 `震` 的测试和构建动作。
- 优先使用瞬时容器；没有容器时可退化为临时 venv，但必须记录降级。
- 统一 timeout。
- 清理进程、容器、临时文件。

输出必须包含：

```text
exit_code / duration_ms / stdout_digest / stderr_digest / truncated_summary
```

### 7.5 `soft_feedback`

职责：

- 执行 `兑` 的失败消化。
- 将测试失败、超时、权限拒绝、扫描风险压缩为短 JSON。
- 不伪造成功。
- 不倾倒海量日志。

反馈格式：

```json
{
  "status": "needs_fix",
  "source_state": "震",
  "next_state": "离",
  "summary": "pytest failed in tests/test_api.py::test_create_note",
  "evidence": {
    "exit_code": 1,
    "line_refs": ["tests/test_api.py:42", "app/main.py:31"]
  }
}
```

### 7.6 `archive_guard`

职责：

- 执行 `坤` 的物理收工。
- 清理临时容器和工作目录垃圾。
- 生成 manifest 和 SHA256。
- 可选只读固化。

只读固化默认不应破坏开发者后续继续编辑。建议分两级：

| 模式 | 行为 |
| --- | --- |
| `audit_only` | 只生成 manifest/hash，不 chmod |
| `lockdown` | `chmod` 只读，用于高安全交付场景 |

MVP 默认使用 `audit_only`，避免本地开发体验被锁死。

## 8. 状态转移门

| 事件 | 当前状态 | 下一个状态 | 条件 |
| --- | --- | --- | --- |
| 需要新建/修改文件 | 任意合法入口 | `乾` | 目标路径在工作区内 |
| 写入完成 | `乾` | `震` | 有 changed files |
| 测试通过 | `震` | `坤` | `exit_code == 0` |
| 测试失败 | `震` | `兑` | `exit_code != 0` 且失败次数未超限 |
| 失败反馈完成 | `兑` | `离` | 已生成短反馈 |
| 上下文卡生成 | `离` | `乾` | 有可修复目标 |
| 越权动作 | 任意状态 | `艮` | 命中路径/命令/密钥/网络硬禁区 |
| 连续失败超限 | `震` / `乾` | `艮` | 默认 3 次 |
| 归档完成 | `坤` | `总` | manifest/hash 生成成功 |
| 低置信度 | 任意入口 | `巽` 或 V1 `问` | 需要用户澄清 |

## 9. 证据标准

Build Mode 中，模型自然语言不能作为完成证据。

最低完成证据：

| 证据 | 来源 |
| --- | --- |
| 文件真实落盘 | `scoped_writer` |
| 路径在作用域内 | `scoped_writer` |
| 测试真实执行 | `sandbox_runner` |
| 退出码 | `sandbox_runner` |
| 失败压缩反馈 | `soft_feedback` |
| manifest/hash | `archive_guard` |
| 最终摘要 | V1 `总` |

完成定义：

```text
任务只有在 changed_files 非空、测试证据存在、exit_code == 0、manifest 生成成功时，才能声称完成。
```

## 10. 和测试工作的关系

本文档不是测试报告，也不声称当前代码已经实现全部规则。

下一轮测试必须等以下最小实现完成后再进行：

1. `intent_resolver` 能把完整项目任务路由到 `乾`。
2. `permission_matrix` 能按状态下发不同工具。
3. `scoped_writer` 能真实写入工作区。
4. `sandbox_runner` 能真实运行 pytest 并回传退出码。
5. `soft_feedback` 能把失败转换为短反馈。
6. `archive_guard` 能生成 manifest/hash。

只有这六件事齐备，才可以重新做“裸 Codex vs 一字诀 Codex”的完整项目构建 A/B。

## 11. 非目标

Build Mode MVP 暂不解决：

- 完整 64 卦全部实现。
- 对任意第三方 CLI 的透明无感劫持。
- 真实生产级 Docker 安全配置全覆盖。
- 多租户权限和远程协作。
- 自动公开发布。

这些应在 Build Mode 最小闭环通过后分阶段处理。
