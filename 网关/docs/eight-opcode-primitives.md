# 八卦原型：一字诀 AgentOS 八大 Opcode

日期：2026-05-24
定位：V0.3 根字架构与 V0.4 内核运行规范
状态：基础字段、继承校验、根字 workflow markdown、workflow loader、Kernel Runtime Policy 和网关注入已写入运行时代码

## 1. 核心判断

一字诀不应该从一开始就硬编码 64 个彼此平级的执行字。更稳的工程路线是：

```text
8 个根字 = AgentOS 底层行为原语
22 个现有字 = 第一批工程化派生字
64 个未来字 = 八个根字的完整扩展指令集
```

八个根字负责定义最稳定、最正交、最难变化的系统行为边界。未来新增任何字，都必须先选择一个根字继承。

子字可以比父字更严格，但不能比父字更宽松。

每个根字背后都必须是一套精准、可靠、稳定、专业的提示词工程和执行规范。根字不是让模型自由解释的符号，而是让网关加载以下内容的 Opcode：

- 专业来源：参考成熟 Agent Skill、调试、TDD、Guardrail、Human-in-the-loop、项目记忆和上下文压缩等优秀工程模式。
- 效率控制：最小上下文、确定性路由、避免重复探索、控制 token 预算。
- 精准控制：事实/推断/未知分离，文件、行号、命令和来源证据绑定。
- 稳定控制：权限继承、失败熔断、上下文断路器、工具守卫。
- 证据要求：系统层 stdout/stderr、exit code、hash、路径和用户确认，而不是模型自称。

补充说明：根字也可以从更底层的阴阳二进制控制位解释。`阳 = Model Layer`，代表大模型的生成、推理和归纳能力；`阴 = System Layer`，代表网关、权限、沙盒、退出码和证据链。详见 [阴阳二进制内核](yin-yang-binary-kernel.md)。运行时仍以 `OneWordState`、`KernelPolicy`、`MutationEngine` 和测试为准。

## 2. 设计原则

### 2.1 八字为骨

八个根字不是业务功能清单，而是 Agent 行为原语。它们对应系统最底层的动作类型：

```text
查 = 观察
修 = 改动
测 = 验证
卫 = 防护
停 = 熔断
问 = 澄清
记 = 记忆
总 = 收束
```

这八个动作足以覆盖大多数 Agent 基本需求。

### 2.2 六十四为展

64 字不是推翻八个根字，而是在八个根字下做专业化变体：

```text
根字定义物理边界
子字定义专业场景
```

例如：

```text
查 -> 解 / 审 / 源 / 搜
修 -> 改 / 简 / 优 / 补
卫 -> 合 / 隔 / 禁 / 守
总 -> 文 / 交 / 归 / 摘
```

### 2.3 字义归模型，解释归文档

八卦、卦爻、变卦在这里不是玄学解释，而是一套工程建模语言：

```text
卦 = 执行状态
爻 = 阶段步骤
变卦 = evidence 驱动的状态转移
```

最终执行权必须在系统层，而不是交给大模型临场解释。

## 3. 八个根字的三维控制向量

每个根字必须在系统内核层定义三维控制向量：

- 权限向量：允许哪些工具、是否能写入、是否能联网、是否能安装依赖。
- 上下文向量：是否清空上下文、是否保留证据、是否触发断路器。
- 证据向量：必须产出什么系统层证据，模型不能伪造。

| 根字 | Opcode | 权限向量 | 上下文向量 | 证据向量 |
| --- | --- | --- | --- | --- |
| `查` | inspect | `Read-Only`，仅读文件、代码、日志和只读命令 | 保留全局上下文，但只读取最小必要范围 | 文件路径、结构树、行号范围、搜索结果 |
| `修` | fix | `Write-Scoped`，只允许修改影响范围内文件 | 一字一清理，丢弃无关中间历史 | 失败复现、修改摘要、测试/构建输出、证据 hash |
| `测` | verify | `Execute-Only`，运行测试、构建、检查命令，默认不改生产代码 | 使用纯净代码输入和上一步产物摘要 | 测试 stdout、exit code、覆盖率、证据 hash |
| `卫` | guard | `Kernel-Filter`，拦截高危命令、依赖安装、越权写入 | 持续监控，安全上下文置顶 | 权限判定、违规原因、安全审查结果 |
| `停` | halt | `Zero-Write / Zero-Risk`，停止写入和高风险工具；必要时停止上游模型调用 | 冻结当前执行栈和状态 | 熔断原因、失败证据、人工恢复条件 |
| `问` | clarify | `UI-Prompt`，只允许向人类请求确认或澄清 | 提取争议焦点，压缩为最小问题上下文 | 用户确认、布尔选择、结构化参数 |
| `记` | record | `Storage-Write`，只写项目记忆、ADR、文档或指定知识库 | 状态压缩归档，禁止写入临时猜测 | 写入文件路径、内容摘要、磁盘 hash |
| `总` | summarize | `Context-Compress`，只做总结、交接、状态压缩 | 清洗并丢弃冗余上下文，保留事实和风险 | Markdown 摘要、下一步建议、来源引用 |

## 4. 根字 Workflow 质量契约

根字 workflow 文件位于：

```text
agent_skill_dictionary/workflows/
```

每个根字必须包含五个质量契约章节：

```text
Prompt Engineering Sources
Efficiency Controls
Precision Controls
Stability Controls
Evidence
```

这五个章节保证一个字背后不是一句泛泛提示词，而是一套可以被测试、加载和注入的专业规范。

| 根字 | 专业提示词工程核心 | 运行目标 |
| --- | --- | --- |
| `查` | Reader-only、按需上下文、证据绑定 | 快速调查，不写入，不猜测 |
| `修` | 系统调试、TDD、外科手术式修改、验证前不宣称完成 | 修复问题，同时控制改动范围 |
| `测` | CI 质量门、真实命令输出、证据优先 | 用系统证据判断通过或失败 |
| `卫` | PreToolUse、Guardrail、零信任权限 | 在工具执行前阻断风险 |
| `停` | Tripwire、人工恢复、失败降级 | 冻结风险流程，收回权限 |
| `问` | 需求澄清、PermissionRequest、Human-in-the-loop | 用最少问题消除关键不确定性 |
| `记` | CLAUDE.md / AGENTS.md、ADR、Docs-as-source-of-truth | 保存稳定项目记忆，不保存猜测 |
| `总` | Context compaction、handoff summary、证据保留 | 压缩上下文，交接事实和下一步 |

网关会按 `root_opcode` 加载这些 workflow 摘要并注入 system rule。派生字不会复制一份根字规范，而是继承根字的权限、上下文、证据和稳定性约束。

## 5. 内核运行策略：三项控制铁律

根字 workflow 负责“专业规范”，Kernel Runtime Policy 负责“物理规训”。每个根字激活后，网关执行三项控制铁律：

1. 工具权限锁：只保留根字允许的工具，其他 `tools` 在请求转发前被过滤。
2. 内核行为规训：注入根字硬 System Prompt，并用根字策略覆盖 `temperature`。
3. 原子证据链：声明必须返回的证据字段，后续可作为交付门和审计日志依据。

当前 MVP 的运行时映射：

| 根字 | 卦象 | 内核名 | 工具权限锁摘要 | 原子证据链 |
| --- | --- | --- | --- | --- |
| `查` | 离 | inspect | `read_file / list_directory / grep_code / git_diff` | `Target_Files_List / Line_Numbers / Log_Context_Snippets` |
| `修` | 震 | fix | `read_file / edit_scoped_file / create_new_file` | `Git_Diff_Patch / Modified_Line_Numbers / Fix_Logic_Explanation` |
| `测` | 巽 | verify | `run_pytest / run_npm_test / capture_coverage` | `Test_Stdout_Log / Coverage_Percentage / Exit_Code` |
| `卫` | 坎 | guard | `dependency_security_scan / ast_vulnerability_check` | `Security_Audit_Report / Risk_Level / Blocked_Evidence` |
| `停` | 艮 | halt | 无工具；HTTP 503 阻断上游模型转发 | `Memory_State_Dump / Failure_Counter / Activation_Token_Status` |
| `问` | 兑 | prompt | `send_user_message / render_ui_options` | `JSON_Choice_Prompt` |
| `记` | 坤 | store | `append_knowledge_base / write_markdown_doc / git_commit` | `Stored_File_Path / SHA256` |
| `总` | 乾 | summarize | `compress_tokens` | `Summary_Markdown / Open_Risk_List / Next_Opcode_Recommendation` |

注意：`造` 在当前工程设计中仍然继承 `修`。它代表构建/新建功能，但不是根字；这样可以避免把“受限写入”和“创造性构筑”拆成两个互相重叠的底层原语。

## 6. 根字与现有 22 字的继承关系

当前 V0.3 词典已有 22 个字，并已补上 `root_opcode`。

| 根字 | 现有派生字 | 继承说明 |
| --- | --- | --- |
| `查` | `解 / 审 / 源 / 搜` | 全部继承只读调查基因，不得写入业务代码 |
| `修` | `改 / 简` | 继承受限写入基因，只能外科手术式改动 |
| `测` | 暂无直接派生 | 后续可扩展覆盖率、回归、压力测试等验证字 |
| `卫` | `合 / 隔` | 继承安全、防护、隔离和合规边界 |
| `停` | 暂无直接派生 | 后续可扩展冻结、回滚、等待审批等控制字 |
| `问` | 暂无直接派生 | 后续可扩展选择、澄清、确认、授权等交互字 |
| `记` | `文` | 继承受限文档/记忆写入，不得修改业务源代码 |
| `总` | 暂无直接派生 | 后续可扩展摘要、交接、归档、复盘等收束字 |

特殊说明：

- `造 / 部 / 数 / 设 / 评` 属于复合字，V0.3 已先为它们选择主根字；后续可继续补辅助根字字段。
- 复合字可以拥有一个主根字和若干辅助根字，但权限必须取更严格的一方。

建议初始归属：

| 字 | 建议主根字 | 辅助根字 | 理由 |
| --- | --- | --- | --- |
| `造` | `修` | `问 / 测` | 新建功能本质是受限写入，但需要澄清和验证 |
| `部` | `卫` | `测 / 停` | 部署是高风险动作，应以安全门为主 |
| `数` | `记` | `测` | 数据处理写入产物，但要保留可复现证据 |
| `设` | `修` | `测 / 记` | UI 改动是受限写入，需要视觉验证和设计记录 |
| `评` | `查` | `卫` | 二次评估默认只读，发现高风险时转安全 |

## 7. 子字继承规则

### 7.1 权限只可收紧

如果父字是只读，子字必须只读。

```text
查 = Read-Only
解 / 审 / 源 / 搜 <= Read-Only
```

子字不能因为自己是“更专业”就扩大权限。

### 7.2 证据只可增加

子字可以增加更具体的证据要求，但不能删除父字必须保留的证据。

例如：

```text
查 要求 source_reference
源 还要 dependency_manifest_hash / license_reference
```

### 7.3 上下文只可更小

子字可以要求更严格的上下文隔离，但不能继承更多无关历史。

### 7.4 风险冲突取更严

如果一个字组合了多个根字，最终权限取更严格的一方。

例如：

```text
部 = 卫 + 测 + 停
```

部署流程可以运行构建验证，但高风险发布必须走 `卫` 的安全门和 `停` 的人工确认。

## 8. 六爻工作流规范

每个根字都可以定义一个六步工作流。六步不是为了美学，而是为了限制 Agent 不跳步。

### 8.1 `查` 的六步

1. 确认调查目标。
2. 定位相关文件、日志、配置或外部来源。
3. 读取最小必要上下文。
4. 区分事实、推断和未知项。
5. 输出证据路径、行号或来源。
6. 推荐下一执行字：`问 / 修 / 卫 / 总`。

### 8.2 `修` 的六步

1. 捕获真实失败上下文。
2. 建立最小复现路径。
3. 定位影响范围。
4. 执行最小必要修改。
5. 运行验证。
6. 成功转 `测` 或 `审`，失败熔断到 `查 / 停`。

### 8.3 `测` 的六步

1. 确认验证对象和通过标准。
2. 选择项目已有测试/构建命令。
3. 运行命令并捕获 stdout、stderr、exit code。
4. 生成证据 hash。
5. 判断通过、失败或证据不足。
6. 成功转 `总 / 记`，失败转 `修 / 查`。

### 8.4 `卫` 的六步

1. 捕获待执行动作或待审对象。
2. 做权限白名单检查。
3. 做危险命令、依赖、网络、路径和密钥风险判断。
4. 允许、拒绝或转人工确认。
5. 记录违规原因和证据。
6. 高风险转 `停`，低风险返回原执行链。

### 8.5 `停` 的六步

1. 冻结当前执行栈。
2. 收回写入、部署、依赖安装和高风险工具权限。
3. 汇总失败证据和触发原因。
4. 给出恢复、降级或终止选项。
5. 等待用户确认。
6. 根据确认恢复到指定执行字或终止。

### 8.6 `问` 的六步

1. 定位缺失信息或冲突意图。
2. 压缩成最少问题。
3. 给出选项和影响。
4. 等待用户输入。
5. 结构化确认结果。
6. 重新编译执行字或恢复执行栈。

### 8.7 `记` 的六步

1. 确认要记录的事实、规则或决策。
2. 检查是否已验证且适合长期保存。
3. 选择写入位置：README、AGENTS、ADR、docs 或知识库。
4. 写入简短结构化内容。
5. 捕获文件 hash。
6. 输出记录位置和后续引用方式。

### 8.8 `总` 的六步

1. 收集已完成事实。
2. 收集未完成事项、风险和失败证据。
3. 丢弃冗余对话和临时推断。
4. 生成结构化摘要。
5. 推荐下一执行字。
6. 输出可交接状态。

## 9. 闭环 Macro Chain 组合规则

复杂需求不应该生成一个巨大执行字，而应编译为根字链。Macro Chain 是一字诀的闭环控制环：每一步只激活一个字，每个字重新加载自己的权限、上下文、工具锁、内核规训和证据要求。

当前代码入口：

```text
agent_skill_dictionary/macro_chain.py
```

### 9.1 功能开发闭环

示例：

```text
用户：帮我写一个高效的异步日志模块，并确保在网络断开时不会丢失日志。
```

建议编译为：

```text
查 -> 造 -> 测 -> 修 -> 记 -> 总
```

闭环含义：

- `查` 只读盘点现有架构、依赖和切入点。
- `造` 作为 `修` 的派生字进入受限构建模式，创建新模块或新接口。
- `测` 运行并发、断网、边界和异常验证。
- `修` 只针对失败测试做外科手术式修复。
- `记` 将通过验证的架构决策和经验写入项目记忆或文档。
- `总` 输出交接摘要、证据和下一步。

### 9.2 安全熔断闭环

示例：

```text
用户或运行过程：检测到脚本里有 rm -rf 和未知外联，先安全熔断并让人确认。
```

建议编译为：

```text
卫 -> 停 -> 问 -> 查 -> 总
```

闭环含义：

- `卫` 检查危险命令、外联、依赖和注入风险。
- `停` 高风险时收回工具并阻断上游模型转发。
- `问` 把风险和恢复选项交给人类确认。
- `查` 降级到只读调查，输出漏洞或风险报告。
- `总` 生成可交接的安全摘要。

### 9.3 当前边界

当前 `/v1/yizijue/resolve` 会返回 Macro Chain，供 UI、调度器或后续 AgentOS 执行器使用。`/v1/chat/completions` 仍执行单步请求重写；真正的多步自动执行器是后续阶段。

## 10. OneWord-Agent FSM 框架层

8 个根字已经不仅是词典分类，也可以作为 Agent 的有限状态机运行。当前框架代码入口：

```text
agent_skill_dictionary/one_word_agent.py
```

框架把 8 个根字映射成 8 个确定状态：

```text
离 = 查
震 = 修
巽 = 测
坎 = 卫
艮 = 停
兑 = 问
坤 = 记
乾 = 总
```

运行时不让模型自己决定“接下来随便做什么”，而是由 `MutationEngine` 按证据转移：

- `修` 成功后进入 `测`。
- `测` 失败后回到 `修`。
- 连续失败超过重试上限进入 `停`。
- 高风险进入 `停`。
- 需要人类确认进入 `问`。
- 完成归档后进入 `总`。

这使一字诀从“网关中间层”继续向“Agent 框架层”推进：网关负责单次请求规训，OneWord-Agent 负责多步任务的状态轨迹、审计日志和熔断边界。

详细说明见：

```text
docs/oneword-agent-framework.md
```

## 11. 与 64 字扩展的关系

64 字不应一次写完。扩展顺序应当是：

```text
先固化 8 个根字
再为现有 22 字补 root_opcode
再挑高频场景扩到 32 字
最后根据真实使用日志扩到 64 字
```

新增子字必须满足：

- 能明确继承一个根字。
- 与同根字下已有子字正交。
- 有独立场景和证据要求。
- 权限不超过父字。
- 能被 macro chain 组合。
