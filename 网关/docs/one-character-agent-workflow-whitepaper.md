# 一字驱动 Agent Skill 词典

基于汉字符号的 AI 工作流、工程规范与安全控制方案

版本：V0.3
日期：2026-05-24
项目：字溯东方 / 大字典

## 1. 摘要

“一字驱动 Agent Skill 词典”是一套面向 AI Agent 的确定性控制方案。它的核心思想是：用户可以用自然语言表达需求，系统先把这段模糊输入归一化为一个预定义的“执行字”；然后由这个字加载对应的 Skill、工程规范、权限边界、工作流步骤、验证要求和失败回退策略。

在这个体系中，一个字不是简单的提示词缩写，也不是让大模型自由发挥的暗号。它是 AI 系统内部的高压缩控制入口。

一个字可以代表：

- 一套软件开发流程。
- 一组安全护栏。
- 一套 UI 设计规范。
- 一种多 Agent 隔离模式。
- 一组工具权限。
- 一套测试和验证标准。
- 一条可审计证据链。

V0.2 以后，每个执行字还必须包含两类可审计字段：

- `reference_workflow_patterns`：它借鉴的社区优秀 Skill、官方 Agent 框架、工程规范或安全模式。
- `professional_protocol`：它真正执行时遵守的专业运行逻辑，包括参考来源、步骤和硬门规则。

因此，本方案的目标不是“让 AI 理解一个字有多深”，而是“让系统规定这个字能触发什么、不能触发什么、必须如何验证”。它把大模型容易漂移的自然语言执行过程，转化为可路由、可约束、可复用、可审计的 Agent 工程流程。

## 2. 背景与问题

当前 AI Agent 已经可以读代码、改代码、运行命令、调用浏览器、生成 UI、处理数据，能力越来越强。但能力越强，风险也越明显。

常见问题包括：

- 用户表达模糊，Agent 自行猜测任务类型。
- Agent 在只应查看时擅自修改文件。
- Agent 为了完成任务大范围重写无关代码。
- Agent 编造不存在的接口、文件、依赖或测试结果。
- 每次任务都需要重复注入大量流程提示词，token 成本高。
- 不同优秀工作流散落在各类 Skill、CLAUDE.md、DESIGN.md、安全规则和项目规范中，难以统一调度。
- 多个任务意图混在一句话中，导致权限和上下文相互污染。
- 长上下文持续膨胀，后续任务容易继承前序任务的错误假设。

例如用户说：

```text
这个项目跑不起来，你帮我看看，顺便优化一下。
```

这句话至少可能包含三个任务：

- `查`：先调查为什么跑不起来。
- `修`：尝试修复错误。
- `改`：优化或重构。

如果没有确定性控制层，Agent 很可能跳过调查，直接修改代码；或者把“顺便优化”扩大成无关重构。

本方案要解决的核心问题是：

```text
如何把用户自然语言的模糊意图，转换为一个可执行、可限制、可验证的 Agent Skill 工作流？
```

## 3. 核心思想

本方案采用“三段式控制”。

第一段：语义归一化
系统把用户自然语言映射为一个执行字，例如 `修`、`审`、`测`、`设`、`卫`。

第二段：词典硬解码
系统根据执行字读取固定定义，而不是让模型临场解释这个字。

第三段：Skill 与工作流执行
系统加载该字绑定的 Skill、流程、权限、工具白名单、验证标准和输出格式。若一次输入被识别为多个执行字，系统不把它们混成一个大 Prompt，而是压入指令堆栈，逐字执行、逐字清理上下文、逐字重载权限。

整体流程如下：

```text
用户自然语言输入
  ↓
语义归一化
  ↓
得到执行字
  ↓
查询 Agent Skill 词典
  ↓
加载 Skill / 工作流 / 权限 / 验证规则
  ↓
压入指令堆栈并路由到对应 Agent 或工具链
  ↓
按栈逐字执行任务
  ↓
捕获系统层验证证据
  ↓
输出可审计结论
```

这里的关键不是“一个字省了几个 token”，而是这个字把大量重复规则、流程判断、权限约束和验证要求从模型上下文中搬到了系统层。

## 4. 与“大字典”的关系

当前“大字典 / 字溯东方”项目已经具备汉字基础数据层，包括现代字义、拼音、部首、笔画、繁体异体、复核标签和 Agent 查询接口。

“一字驱动 Agent Skill 词典”可以作为该项目的第二条主线：

- 原有字典回答“这个字在语言和文化中是什么意思”。
- 新词典回答“这个字在 AI 系统中触发什么能力”。

也可以理解为：

```text
解释型字典 → 面向人，解释字义
执行型字典 → 面向 Agent，约束行为
```

解释型字典关注字源、字义、音韵、用法和历史。执行型字典关注 Skill、工作流、工具权限、模型参数、安全边界和验证证据。

这使“大字典”从汉字知识库进一步演进为 AI Agent 的语义控制基础设施。

## 5. 核心定义

### 5.1 执行字

执行字是一个被系统预定义的汉字或符号，用于触发确定的 Agent 能力。

执行字必须满足：

- 含义写死。
- 权限写死。
- 工作流写死。
- 验证要求写死。
- 失败回退策略写死。
- 上下文切换策略写死。
- 证据捕获方式写死。

执行字可以被人类记忆和使用，但它的最终含义由系统词典决定。

### 5.2 Skill

Skill 是可复用的能力单元，可以来自项目内部，也可以参考社区优秀实践。它可以是一段流程说明、一个工具调用规范、一个安全策略、一套设计系统说明，或一个多 Agent 协作模式。

本方案不建议把执行字直接绑定到某个固定 GitHub 仓库。更好的方式是绑定“能力模式”，再把具体仓库作为候选实现或参考来源。

例如：

```text
卫 ≠ 某一个安全项目
卫 = 安全防护能力模式
```

这样即使外部项目更新、下架或不再适合，执行字的核心定义仍然稳定。

### 5.2.1 参考模式与专业协议

执行字不能只写一句“做某事”。它必须把优秀社区实践拆成可执行字段：

```json
{
  "reference_workflow_patterns": [
    "superpowers:systematic-debugging",
    "openai-agents:tool-guardrail",
    "langgraph:human-in-the-loop"
  ],
  "professional_protocol": {
    "source_projects": ["具体参考项目或官方文档"],
    "operating_logic": ["进入该字后的专业步骤"],
    "hard_gates": ["必须阻断、追问、降级或熔断的条件"]
  }
}
```

这让一字诀借鉴 Claude Agent Skills、OpenAI Skills、Vercel Agent Skills、Superpowers、Claude Code Hooks、OpenAI Agents SDK Guardrails、LangGraph human-in-the-loop、MCP、SWE-agent、DESIGN.md 和安全社区实践时，不是简单复制仓库，而是抽象它们的工程精髓。

### 5.3 工作流

工作流是执行字触发后的确定性步骤。例如 `修` 的工作流可以是：

```text
复现问题 → 定位根因 → 最小修改 → 补测试 → 运行验证 → 汇报证据
```

### 5.4 权限边界

权限边界规定 Agent 能做什么、不能做什么。例如：

- `查` 可以读文件，不能写文件。
- `审` 可以输出审查意见，不能直接修改。
- `修` 可以编辑相关文件，但不能做无关重构。
- `卫` 可以阻断危险操作，必须要求人工确认高风险行为。

### 5.5 验证标准

验证标准规定 Agent 如何证明任务完成。例如：

- 测试输出。
- 构建结果。
- 复现结果。
- 文件和行号。
- 安全扫描结论。
- 人工确认记录。

没有验证证据，就不能声称任务完成。

### 5.6 指令堆栈

指令堆栈用于处理多字组合。多个执行字不能简单拼接成一个混合提示词，而应按顺序压入栈中，逐字执行。

例如 `修 + 测` 表示先修复再测试。系统可以将执行顺序压栈为：

```text
Stack.push(测)
Stack.push(修)
```

执行层先弹出 `修`，加载 `修` 的权限、Skill、工具链和上下文；`修` 完成并产出证据后出栈。然后系统弹出 `测`，重新加载 `测` 的最小上下文和工具权限。此时写权限可以被锁死，只保留测试运行与结果读取权限。

指令堆栈的目标是防止多字组合时发生语义污染、权限继承和上下文漂移。

### 5.7 上下文预算断路器

上下文预算断路器用于控制长流程中的上下文膨胀。每次执行字切换时，系统应默认清理中间对话历史，只保留：

- 原始需求。
- 当前执行字所需的最小项目上下文。
- 上一个执行字产出的结构化结果。
- 系统层捕获的验证证据摘要。

这使 `造` 转 `测`、`修` 转 `审` 等流程不会携带过量历史，也减少 token 成本和逻辑惯性。

### 5.8 降级与熔断

降级与熔断用于处理连续失败的工作流。如果某个执行字连续失败、回退或验证不通过超过设定次数，系统应收回高风险权限，将当前任务降级到更安全的执行字。

例如 `修` 连续失败 3 次后，系统应触发熔断：

```text
修 → 查
```

此时写权限关闭，系统整理失败证据、疑似根因、已尝试方案和下一步建议，等待人类确认。

## 6. 系统总体架构

系统分为八层。

### 6.1 输入层

输入层接收用户自然语言、代码、文件、日志、图片、链接或结构化数据。

输入层不直接决定任务类型，只负责保留原始上下文。

### 6.2 语义归一化层

语义归一化层把用户输入映射为执行字。

推荐采用混合策略：

- 显式命令优先：用户直接写 `修：`、`审：`、`设：` 时直接采用。
- 关键词规则：高频表达用规则命中。
- 轻量分类模型：输出执行字和置信度。
- 向量检索：用真实表达样本辅助判断。
- 低置信度追问：不确定时先问清楚。

这一层可以使用模型，但它只负责分类，不负责最终执行。

### 6.3 词典硬解码层

词典硬解码层根据执行字读取固定配置，包括：

- 系统定义。
- 绑定 Skill。
- 工作流步骤。
- 允许工具。
- 禁止工具。
- 写权限范围。
- 模型策略。
- 输出格式。
- 验证要求。
- 失败回退。

这一层应由 JSON、数据库、代码常量或工作流引擎实现，不依赖大模型自由解释。

### 6.4 Skill 加载层

Skill 加载层负责加载该字对应的能力模块。

例如：

- `修` 加载调试、TDD、验证 Skill。
- `造` 加载 Spec、计划、实现、测试 Skill。
- `设` 加载 DESIGN.md、组件规范、视觉一致性 Skill。
- `卫` 加载安全策略、危险操作拦截、依赖审查 Skill。
- `隔` 加载 Reader / Orchestrator / Writer 隔离模式。

Skill 加载应遵循最小上下文原则：只加载当前执行字需要的规则，不把所有 Skill 一次性塞给模型。

### 6.5 指令堆栈层

指令堆栈层负责处理多字组合。它决定执行顺序、权限重载、上下文清理和失败回退。

执行原则：

- 一个执行字一次只加载自己的 Skill 和权限。
- 每个执行字完成后必须产出结构化结果或失败证据。
- 切换到下一个执行字时触发上下文预算断路器。
- 后一个执行字不能继承前一个执行字的高风险权限，除非词典明确允许。

### 6.6 路由层

路由层根据执行字选择具体 Agent、工具链或工作流处理器。

例如：

```text
修 → debug_fix_workflow
审 → code_review_workflow
设 → design_system_workflow
卫 → security_guard_workflow
隔 → isolated_multi_agent_workflow
```

路由层应尽量由确定性代码执行，而不是让模型自己决定。

### 6.7 执行层

执行层调用实际能力，包括：

- 读取文件。
- 搜索代码。
- 修改文件。
- 运行测试。
- 调用浏览器。
- 调用数据库。
- 调用外部 API。
- 调用静态分析工具。
- 调用多 Agent 子流程。

执行层必须受权限边界约束。

### 6.8 验证与审计层

验证与审计层负责记录：

- 识别出的执行字。
- 加载的 Skill。
- 使用的工具。
- 修改的文件。
- 运行的验证命令。
- 失败或回退原因。
- 最终证据。
- 证据来源。
- 命令退出码。
- 时间戳。
- stdout / stderr 摘要。
- 证据摘要哈希。

这层是降低幻觉、支持复盘和建立信任的关键。证据应尽量由底层运行环境、沙盒或工具调用系统直接捕获，而不是由模型手写。模型可以读取并解释证据，但不应拥有篡改审计日志的权限。

## 7. 第一批执行字设计

第一版建议从编程和工程规范场景开始，因为它们最容易验证。

| 字 | 能力名称 | 核心含义 | 绑定能力模式 |
| --- | --- | --- | --- |
| `查` | Inspect | 只读调查项目、文件、结构、原因 | 只读探索、上下文收集 |
| `解` | Explain | 解释代码、报错、架构或概念 | 文档解释、结构化讲解 |
| `修` | Fix | 修复 bug、失败测试或运行异常 | 系统调试、TDD、验证 |
| `造` | Build | 新增功能、脚本、接口或组件 | Spec、计划、实现、测试 |
| `改` | Change | 修改、优化或重构已有实现 | 外科手术式修改、保持接口 |
| `测` | Test | 编写、补充或运行测试 | 单测、集成测试、回归验证 |
| `审` | Review | 代码审查，只找问题和风险 | 只读 review、证据优先 |
| `设` | Design | UI 与设计系统实现 | DESIGN.md、视觉规范、一致性 |
| `源` | Source | 依赖、License、本地已有能力和代码资产溯源 | 依赖审计、来源检查、复用优先 |
| `卫` | Guard | 安全防护与危险操作拦截 | 权限白名单、依赖审查、高危阻断 |
| `隔` | Isolate | 多 Agent 隔离协作 | Reader / Orchestrator / Writer |
| `简` | Simplify | 极简、克制、外科手术式改动 | 简单优先、少封装、低扩散 |

其中 `源` 与 `卫` 的边界需要明确区分：

- `源` 负责来源审查：依赖来源、License、已有代码资产、重复造轮子风险。
- `卫` 负责安全防护：危险命令、越权写入、提示词注入、未批准依赖安装。

这些字不必一次全部实现。MVP 可以先做 `查 / 修 / 审 / 测 / 造`，再扩展 `设 / 源 / 卫 / 隔 / 简`。

## 8. 执行字与优秀工作流的关系

用户收集的社区项目和规范可以进入本方案，但应以“能力模式”方式吸收。

### 8.1 工程闭环能力

代表方向：

- 构思。
- 编写 Spec。
- 拆解任务。
- 测试驱动开发。
- 人工 Review。
- 验证后合并。

可绑定执行字：

- `造`：新增功能时调用完整工程闭环。
- `修`：修复 bug 时调用调试、TDD、验证闭环。
- `测`：专门调用测试与验证闭环。

### 8.2 防卫性编程规范

代表方向：

- 修改前先思考。
- 简单优先。
- 外科手术式修改。
- 严禁大面积无关重写。
- 每次改动尽量有测试兜底。

可绑定执行字：

- `简`：强制采用最小改动策略。
- `改`：重构或优化时限制扩散。
- `修`：修 bug 时避免顺手重写。

### 8.3 UI 与设计系统规范

代表方向：

- DESIGN.md。
- 品牌视觉规则。
- 颜色、字体、间距、组件状态。
- 响应式与可访问性。

可绑定执行字：

- `设`：加载设计系统与 UI 实现流程。
- `审`：做 UI 一致性审查时只读检查。

### 8.4 安全防护能力

代表方向：

- 技能来源审查。
- 依赖安装审查。
- 危险命令拦截。
- AST 或规则层面的高危 API 检测。
- 人工批准高风险操作。

可绑定执行字：

- `卫`：安全守卫模式。
- `审`：安全审查模式。
- `隔`：隔离不可信输入和执行权限。

### 8.5 来源与依赖审计能力

代表方向：

- 检查第三方依赖来源。
- 检查 License 与商业使用风险。
- 检索本地已有组件、函数、工具和服务。
- 防止重复造轮子。
- 防止为了完成任务随意新增依赖。

可绑定执行字：

- `源`：来源审查与依赖审计。
- `卫`：在来源不可信或依赖高风险时拦截执行。
- `简`：优先复用已有能力，避免引入复杂新抽象。

典型组合：

```text
源 + 卫 = 先审查来源与依赖，再决定是否允许执行、安装或调用
源 + 简 = 先查本地已有能力，再决定是否需要新增代码
```

### 8.6 多 Agent 隔离能力

代表方向：

- Reader 只读外部内容。
- Orchestrator 只做调度。
- Writer 只在干净上下文中写入。

可绑定执行字：

- `隔`：启用隔离式多 Agent 协作。
- `卫`：对高风险输入启用隔离。
- `审`：审查不可信内容时隔离执行权。

## 9. Token 优化机制

本方案可以显著降低 token 用量，但关键不在于“一个字本身很短”，而在于把重复规则系统化。

### 9.1 减少重复提示词

原来每次任务都要注入完整流程：

```text
请先分析需求，再写计划，避免大范围重构，遵循 TDD，修改后运行测试，不要删除用户代码，输出验证证据……
```

现在可以由 `修` 或 `造` 触发固定配置。模型只接收当前任务需要的最小上下文。

### 9.2 减少意图判断成本

一旦输入被映射为执行字，后续流程无需反复让模型判断“我应该做哪类任务”。

### 9.3 减少上下文污染

`设` 只加载 UI 设计规则，`卫` 只加载安全规则，`修` 只加载调试规则。系统不需要把所有规范同时塞进上下文。

### 9.4 上下文预算断路器

每次执行字切换时，系统应触发上下文预算断路器。断路器负责清空不必要的中间对话、临时推理和过期假设，只保留原始需求、当前执行字所需上下文、上一步结构化输出和系统证据摘要。

例如：

```text
造 → 测
```

`造` 阶段结束后，`测` 不应继承完整开发过程对话。它只需要读取最终变更、测试目标、必要配置和 `造` 阶段产出的证据摘要。这样可以节省 token，也能防止测试阶段被开发阶段的错误假设污染。

### 9.5 减少返工

错误路由、越权修改、未验证结论都会造成大量来回沟通。执行字通过边界和验证减少返工，这往往比压缩提示词本身节省更多 token。

### 9.6 系统层执行

路由、权限判断、工具白名单、失败回退等逻辑应尽量由代码执行，而不是每次都让模型阅读并理解。

## 10. 减少幻觉的机制

执行字体系可以降低幻觉率，但不能承诺绝对消灭幻觉。它的价值是把幻觉问题从“模型自由发挥”转移到“系统规则是否足够明确”。

### 10.1 意图与执行分离

模型可以参与理解用户意图，但执行阶段由词典、Skill 和工作流约束。

### 10.2 字义硬编码

`审` 只能审查，不能修改。`查` 只能调查，不能写文件。`修` 必须定位和验证。含义由系统定义，不由模型临场发挥。

### 10.3 权限最小化

每个字只开放必要工具。例如 `查` 不开放写权限，`卫` 可以阻断依赖安装，`隔` 可以限制不同 Agent 的上下文和权限。

### 10.4 验证优先

涉及代码、构建、测试、安全和数据处理的任务，必须给出工具证据。没有证据的结论只能标记为推断。

证据应由系统层捕获。对于测试、构建、静态分析和命令执行，审计层应记录 stdout / stderr 摘要、退出码、时间戳和 SHA-256 摘要哈希。模型只能读取和转述这些证据，不能自行伪造验证日志。

### 10.5 低置信度回退

当无法确定用户意图时，系统不强行执行，而是追问或回退到只读 `查`。

### 10.6 不可信输入隔离

对于外部网页、第三方文档、用户上传内容和依赖脚本，应使用 `隔` 或 `卫` 模式，避免提示词注入和越权执行。

## 12. 语义归一化策略

执行字体系的关键难点不是词典本身，而是把用户输入稳定映射到正确的字。

### 12.1 明确命令直达

用户可以直接使用执行字：

```text
修：这个测试失败了
审：帮我 review 这次改动
设：把这个页面按 DESIGN.md 统一一下
卫：检查有没有高危依赖和危险命令
```

### 12.2 关键词规则

用规则覆盖高频表达：

- “报错、失败、bug、跑不通、异常” → `修`
- “解释、看不懂、什么意思、讲一下” → `解`
- “新增、实现、做一个、写一个” → `造`
- “优化、重构、调整、改成” → `改`
- “测试、覆盖率、单测、验证” → `测`
- “review、审查、有没有问题、风险” → `审`
- “界面、样式、设计、视觉、组件” → `设`
- “依赖、来源、License、已有组件、复用、不要重复造轮子” → `源`
- “安全、权限、危险命令、依赖、注入” → `卫`
- “隔离、不可信、外部输入、多 Agent” → `隔`
- “简单点、别大改、最小改动” → `简`

### 12.3 分类模型

当规则无法确定时，使用轻量分类模型。模型只输出结构化结果：

```json
{
  "code": "修",
  "confidence": 0.86,
  "secondary_codes": ["测"],
  "reason": "用户描述了运行失败，并要求确认结果。"
}
```

执行层只消费 `code`、`confidence` 和 `secondary_codes`，不把 `reason` 当作事实。

### 12.4 多字组合

复杂任务可以由多个执行字组合，但必须有主次顺序。多字组合必须通过指令堆栈执行，而不是把多个字的规则混合到一个上下文中。

例如：

```text
修 + 测 = 先修复，再验证
设 + 审 = 做 UI 一致性审查，不直接改
源 + 卫 = 先审查依赖与来源，再执行安全拦截
卫 + 隔 = 高风险不可信输入，启用隔离安全模式
造 + 简 + 测 = 新增功能，但保持最小实现并测试
```

多字组合不能绕过权限。比如 `审 + 修` 必须明确用户是否允许修改，否则默认只审查。

### 12.5 指令堆栈调度

当归一化层输出多个执行字时，系统应把它们转换为指令堆栈。

例如用户输入：

```text
这个 bug 修一下，然后跑测试确认。
```

归一化结果：

```text
修 + 测
```

调度方式：

```text
Stack.push(测)
Stack.push(修)
```

执行顺序：

1. 弹出 `修`，加载调试修复 Skill、受限写权限和验证要求。
2. `修` 完成后，系统捕获证据，写入审计日志并出栈。
3. 触发上下文预算断路器，只保留原始需求、最终改动摘要和系统证据摘要。
4. 弹出 `测`，重新加载测试 Skill 和测试工具链。
5. `测` 阶段默认不继承 `修` 的写权限，除非词典明确允许。

这套机制保证每个字都是原子执行单元。它可以组合，但不能混权、混上下文、混责任。

### 12.6 低置信度追问

当置信度不足，或一句话同时命中多个互斥执行字时，系统应追问：

```text
你希望我先只读调查，还是直接尝试修复？
```

追问是安全机制，不是低效。

## 14. 风险与边界

### 14.1 外部项目真实性与可用性风险

社区项目、仓库和规范会变化。本方案不应把核心能力绑定到单一外部项目，而应抽象为能力模式，并在落地前审查许可证、安全性、维护状态和适配成本。

### 14.2 误分类风险

如果用户输入被映射到错误执行字，系统会稳定地执行错误流程。因此必须保留置信度、追问、人工确认和只读回退。

### 14.3 过度压缩风险

一个字可以确定任务类型和控制规则，但不能替代任务上下文。原始输入、文件、日志和业务约束仍需保留。

### 14.4 字义冲突风险

同一个汉字可能同时拥有文化语义、自然语言语义和系统执行语义。执行字的系统含义只在特定 Agent 环境中生效。

### 14.5 规则僵化风险

确定性会提升可靠性，但过度固定可能降低灵活性。解决方式是在执行字下保留参数槽，例如目标文件、技术栈、优先级、风险级别和验证方式。

### 14.6 安全错觉风险

有了 `卫` 或 `隔` 不代表系统绝对安全。安全能力必须落实到工具权限、命令拦截、依赖审查、审计日志和人工确认。

### 14.7 证据伪造风险

如果验证证据由模型自己生成，就仍然可能出现虚假测试日志、虚假构建结果或错误摘要。系统应尽量由运行环境直接捕获 stdout、stderr、退出码、时间戳和摘要哈希，并限制模型修改审计日志的权限。

### 14.8 权限污染风险

多字组合时，如果多个执行字共享同一个上下文和工具权限，后续任务可能继承前序任务的高风险权限。指令堆栈必须在每个字执行前重新加载权限策略，并在切换时触发上下文预算断路器。

### 14.9 熔断误伤风险

熔断机制可能在复杂任务中提前中止合理尝试。为避免误伤，熔断策略应保留人工重新激活入口，并输出清晰的失败证据、已尝试步骤和建议下一步。

## 15. 未来愿景

“一字驱动 Agent Skill 词典”的长期目标，是建立一套面向 AI Agent 的汉字符号操作系统。

在这个系统中，汉字不只是文本内容，而是可执行的智能控制单元：

- 一个字定义一种任务类型。
- 一个字加载一套 Skill。
- 一个字限制一组权限。
- 一个字触发一种工作流。
- 一个字要求一类验证证据。
- 一个字在指令堆栈中作为原子步骤执行。

当这种体系成熟后，“大字典”可以从汉字知识库演进为 AI Agent 的行为基础设施。

最终形态可以是：

```text
人用自然语言表达目标
  ↓
系统归一化为执行字
  ↓
执行字压入指令堆栈
  ↓
逐字加载 Skill、权限与规则
  ↓
Agent 在权限边界内行动
  ↓
工具和测试提供系统层证据
  ↓
系统输出可审计结果
```

这条路线把汉字的高压缩语义能力，与现代 AI Agent 的工程可控性结合起来。它不是让 AI 更自由地想象，而是让 AI 更稳定地执行。

## 16. 结论

本方案的核心价值在于：用一个字，把用户模糊输入压缩成系统可执行、可约束、可验证的 Agent Skill 工作流。

真正提高效率、准确度、降低幻觉和减少 token 的，不是一个字本身，而是这个字背后绑定的固定定义、优秀 Skill、权限护栏、指令堆栈、最小上下文加载、系统证据捕获和验证机制。

因此，“一字驱动 Agent Skill 词典”不是提示词缩写表，而是 AI Agent 的确定性控制层。

编程场景是第一试点。通过 `查 / 修 / 审 / 测 / 造 / 改 / 简 / 设 / 源 / 卫 / 隔` 等执行字，可以逐步建立从开发、测试、设计、依赖溯源、安全到多 Agent 协作的统一控制系统。

一旦该体系跑通，“大字典”将从解释汉字的知识项目，升级为用汉字组织和约束 AI 行为的 Agent 操作系统。
